# 02.3 OSS 复用策略评估

> 本文档评估 AgentFlow 对各开源组件的复用策略、集成深度与长期维护方案。

---

## 复用策略总览

AgentFlow 采用 **"核心自研 + 关键复用"** 策略，将系统组件分为三类：

| 类别 | 策略 | 包含组件 |
|------|------|---------|
| **自研 🟥** | 完全自主研发，不依赖外部 OSS | L0 源编排、L1 Compiler 逻辑、L1 Orchestrator 逻辑、结果汇聚引擎 |
| **复用 🟦** | 直接复用成熟 OSS 组件，在其上层封装业务逻辑 | React Flow、Claude Code CLI、Hermes Agent |
| **集成 🔗** | 通过 API/CLI 调用外部 OSS，松耦合集成 | Docker Engine、JSON Schema 验证器 |

三类策略的选择依据：

```
问题难度 / 领域特异性
        │
   高   │  自研 (核心调度、编排元模型)
        │
   中   │  集成 (Docker、Schema 验证)
        │
   低   │  复用 (UI 框架、LLM CLI、Agent 框架)
        └──────────────────────────► OSS 成熟度
                    低                高
```

## 组件评估明细

### 1. React Flow 🟦 复用

| 维度 | 评估 |
|------|------|
| **版本** | React Flow v12+ (xyflow) |
| **开源协议** | MIT |
| **用途** | L4 画布 UI 的可视化工作流编辑器核心 |
| **集成深度** | 中 — 作为 UI 框架嵌入，上层封装自定义节点类型和业务组件 |
| **替代方案** | DeltaXML 工作流编辑器、自研 Canvas |
| **选型理由** | React Flow 是 React 生态中最成熟的工作流/流程图编辑库，API 稳定，社区活跃；节点/边定制能力强，支持自定义节点渲染和交互事件 |
| **风险** | 复杂布局场景（大量节点嵌套）可能性能下降 |
| **缓解措施** | ① 使用虚拟化（`react-window`）渲染大量节点 ② 限制画布节点的最大数量 |

**复用边界：**

```
React Flow 负责：
├── 节点渲染引擎（NodeRenderer）
├── 边渲染引擎（EdgeRenderer）
├── 拖拽 & 连线交互（DragHandler, ConnectionLine）
├── 布局 & 缩放（PanZoom, Minimap）
└── 事件系统（onNodesChange, onEdgesChange, onConnect）

AgentFlow 封装层负责：
├── 业务节点类型（AgentNode / ToolNode / ConditionNode）
├── 节点配置面板（NodeConfigPanel）
├── 执行状态展示（ExecutionStatusBadge / ProgressBar）
├── 运行时数据绑定（bindNodeState）
└── 工作流导入/导出（WorkflowJSON ↔ React Flow 状态）
```

### 2. Claude Code CLI 🔗 集成

| 维度 | 评估 |
|------|------|
| **版本** | Claude Code (latest) |
| **开源协议** | Anthropic 专有（可通过 CLI 使用） |
| **用途** | Stage 2 Prompt Compiler Agent 的底层引擎 + Stage 4 每个 Agent 节点的执行引擎 |
| **集成深度** | 浅 — 通过 `claude --print` CLI 子进程调用，输入/输出为 JSON 文件 |
| **替代方案** | OpenAI Codex CLI、直接调用 Anthropic API、其他 LLM 后端 |
| **选型理由** | Claude Code 在代码理解、工具调用、长上下文处理方面表现优异；`--print` 模式天然适配工作流节点的非交互式执行；与 Agent 集群的 Docker 化部署高度契合 |
| **风险** | ① API 调用成本（Token 消耗随工作流复杂度线性增长）② 外部依赖（Anthropic API 可用性） ③ CLI 接口变更 |
| **缓解措施** | ① 缓存机制：相同输入跳过重复执行 ② 重试机制 + 退避策略 ③ 版本锁定：固定 Claude Code CLI 版本 |

**集成模式：**

```
┌──────────────────────────────────────────┐
│             AgentFlow 系统                │
│                                           │
│  Prompt Compiler ──JSON──► claude --print │
│  Orchestrator    ──JSON──► claude --print │
│                                           │
│  所有交互通过 标准输入/输出 + 文件         │
│  不依赖 Claude Code 的内部 API            │
└──────────────────────────────────────────┘
```

**未来解耦计划：**

AgentFlow 设计了抽象层 `AgentBackend`，使得 Claude Code 可以替换为其他后端：

```
interface AgentBackend {
  execute(prompt: PromptTask, context: ExecutionContext): Promise<Envelope>;
}

class ClaudeCodeBackend implements AgentBackend { ... }
class OpenAIBackend implements AgentBackend { ... }
class CustomBackend implements AgentBackend { ... }
```

### 3. Hermes Agent 🟦 复用

| 维度 | 评估 |
|------|------|
| **版本** | Hermes Agent (Nous Research) |
| **开源协议** | Apache 2.0 |
| **用途** | Stage 3 Orchestrator 的核心框架 |
| **集成深度** | 中 — 复用 Hermes Agent 的任务调度、工具调用、多步骤执行能力，扩展工作流编排的编排逻辑 |
| **替代方案** | LangChain、AutoGPT、CrewAI、自研编排器 |
| **选型理由** | Hermes Agent 与 AgentFlow 同属 Nous Research 生态，架构理念一致；Hermes Agent 的 skill 系统和 tool-use 机制可以直接映射为工作流节点；作为 Agent 框架本身具备编排能力，减少重复开发 |
| **风险** | Hermes Agent 仍处于快速迭代期，API 可能不稳定 |
| **缓解措施** | ① 在 Orchestrator 中封装适配层（`HermesAdapter`），隔离 Hermes API 变更 ② 贡献上游代码，确保兼容性 |

**集成架构：**

```
┌──────────────────────────────────────┐
│        Orchestrator (自研编排逻辑)      │
│                                      │
│  ┌────────────────────────────────┐  │
│  │   Workflow Scheduler            │  │ ← 自研：工作流拓扑调度
│  │   依赖解析 / 并行决策 / 重试管理  │  │
│  └──────────────┬─────────────────┘  │
│                 │                    │
│  ┌──────────────▼─────────────────┐  │
│  │   Hermes Agent Core (复用)      │  │ ← OSS 复用
│  │   Skill 系统 / Tool 系统        │  │
│  │   多步骤执行 / 上下文管理        │  │
│  └──────────────┬─────────────────┘  │
│                 │                    │
│  ┌──────────────▼─────────────────┐  │
│  │   Docker Runner (集成)          │  │ ← 集成：调用 Docker API
│  │   容器生命周期管理                │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

### 4. Docker Engine 🔗 集成

| 维度 | 评估 |
|------|------|
| **版本** | Docker Engine v24+ |
| **开源协议** | Apache 2.0 |
| **用途** | Stage 4 Agent 集群的容器化运行环境 |
| **集成深度** | 浅 — 通过 Docker SDK/CLI 管理容器生命周期 |
| **替代方案** | Podman、containerd 直接调用、Kubernetes Jobs |
| **选型理由** | Docker 是容器运行时的事实标准，生态成熟；Agent 工作负载天然适合 Docker 的单次运行模式（run → exec → rm）；团队后续也可平滑迁移到 Kubernetes |
| **风险** | ① Docker Daemon 资源竞争 ② 镜像拉取延迟 ③ 安全性（容器逃逸） |
| **缓解措施** | ① 限制并行容器数 ② 预拉取镜像 + 镜像缓存 ③ 使用 `--security-opt` 限制容器权限 |

---

### 5. JSON Schema 验证器 🔗 集成

| 维度 | 评估 |
|------|------|
| **版本** | Ajv (JS) / jsonschema (Python) |
| **开源协议** | MIT / Apache 2.0 |
| **用途** | Stage 5 的 Envelope 输出 Schema 验证 |
| **集成深度** | 浅 — 工具库直接调用 |
| **选型理由** | JSON Schema 是 Schema 验证的行业标准，工具链成熟可靠 |
| **风险** | 极低（成熟工具库） |

## 复用 vs 自研决策树

AgentFlow 在组件选型时遵循以下决策树：

```
该功能是 AgentFlow 的核心竞争力？ ──Yes──→ 自研 🟥
       │
       No
       ▼
OSS 市场有成熟可信的解决方案？ ──Yes──→ 方案成熟且协议兼容？ ──Yes──→ 复用 🟦
       │                              │
       │                              No
       │                              ▼
       │                          评估替代方案或自研
       │
       No
       ▼
该功能可通过标准 API/CLI 调用 OSS？ ──Yes──→ 集成 🔗
       │
       No
       ▼
自研 🟥
```

## 总结：OSS 复用矩阵

| 组件 | 策略 | 协议 | 关键技术依赖 | 解耦准备 |
|------|------|------|-------------|---------|
| React Flow | 🟦 复用 | MIT | 画布渲染、交互 | 封装适配层，可替换为 DeltaXML 等 |
| Claude Code CLI | 🔗 集成 | 专有 | LLM 推理、工具调用 | `AgentBackend` 抽象接口 |
| Hermes Agent | 🟦 复用 | Apache 2.0 | 编排调度、Tool Use | `HermesAdapter` 适配层 |
| Docker Engine | 🔗 集成 | Apache 2.0 | 容器管理 | 抽象 `ContainerRuntime` 接口 |
| JSON Schema | 🔗 集成 | MIT | Schema 验证 | 直接依赖，无解耦必要 |
| React | 🟦 复用 | MIT | UI 框架 | 直接依赖，生态事实标准 |
| TypeScript | 🟦 复用 | Apache 2.0 | 编程语言 | 直接依赖，无解耦必要 |

## 长期维护策略

1. **版本锁定** — 所有 OSS 依赖锁定 `package.json` / `requirements.txt` 的具体版本，通过 Dependabot / Renovate 定期升级
2. **抽象层隔离** — 对关键 OSS 组件（React Flow、Hermes Agent、Claude Code）使用适配器模式封装，降低替换成本
3. **上游贡献** — 对 Hermes Agent 等关键依赖，积极参与上游社区贡献，确保 AgentFlow 需求被纳入上游路线图
4. **降级方案** — 每个 OSS 组件至少有一个降级方案/替代品评估，以应对上游停止维护或协议变更等风险
5. **CVE 监控** — 对所有 OSS 依赖启用安全漏洞监控（GitHub Dependabot / Snyk），及时修复安全风险
