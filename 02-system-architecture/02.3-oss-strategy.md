# 02.3 OSS 复用策略评估

> 本文档评估 AgentFlow 对各开源组件的复用策略、集成深度与长期维护方案。

---

## 复用策略总览

AgentFlow 采用 **"核心自研 + 关键集成"** 策略，将系统组件分为两类：

| 类别 | 策略 | 包含组件 |
|------|------|---------|
| **自研 🟥** | 完全自主研发，不依赖外部 OSS | L0 源编排、Prompt Compiler、DAG 并行执行引擎、AgentRunner、CSS Grid 画布、结果汇聚引擎 |
| **集成 🔗** | 通过标准 API 调用外部服务，松耦合集成 | GLM-5-Turbo API、其他 OpenAI 兼容 LLM API |

策略选择依据：

```
问题难度 / 领域特异性
        │
   高   │  自研 (核心调度、编排元模型、DAG 执行引擎、Compiler)
        │
   中   │  
        │
   低   │  集成 (LLM API 调用)
        └──────────────────────────► OSS 成熟度
                    低                高
```

### 为什么选择"全自研 + API 集成"？

AgentFlow 的核心差异化在于**从用户一句话到完整工作流的自动编排**以及**DAG 并行执行引擎**。这两部分都是高度领域特化的核心竞争力，市场上没有可直接复用的 OSS 组件。因此我们选择全自研核心逻辑，仅通过标准 API 集成外部 LLM 服务。

## 组件评估明细

### 1. AgentRunner 🟥 自研

| 维度 | 评估 |
|------|------|
| **版本** | v1.0 (自研, `agent_runner.py`) |
| **开源协议** | 私有 |
| **用途** | L1 核心引擎的核心执行组件，驱动每个 Agent 节点的 LLM 调用 |
| **集成深度** | 全自研 — 包含 OpenAI 兼容 API 调用、工具调用循环、结果解析 |
| **替代方案** | Claude Code CLI、OpenAI Codex CLI、LangChain Agent |
| **选型理由** | ① 需要支持 12+ 个不同 LLM 供应商的统一抽象 ② 需要轻量级、零容器开销的执行方式 ③ 需要灵活的工具调用循环（bash/read/write）④ Claude Code CLI 和 LangChain 都过于重量级且不够灵活 |

### 2. 多 Provider 抽象层 🟥 自研

| 维度 | 评估 |
|------|------|
| **版本** | v1.0 (自研, 内置于 `agent_runner.py`) |
| **开源协议** | 私有 |
| **用途** | 统一管理所有 LLM API 供应商的认证、端点、模型映射 |
| **集成深度** | 全自研 — 作为 AgentRunner 的内置模块 |
| **替代方案** | LangChain model I/O、LiteLLM、OpenRouter |
| **选型理由** | ① 需要极轻量级的抽象层，不引入 LangChain 等重量级依赖 ② 需要精确控制每个 provider 的 API 端点和 Key 管理 ③ LiteLLM 虽好但增加外部依赖和版本锁定风险 |
| **风险** | ① 供应商 API 变更（端点/协议）② API Key 管理安全 |
| **缓解措施** | ① 供应商适配统一使用 OpenAI 兼容格式，大部分 API 变更只影响 base_url ② API Key 通过环境变量注入，不硬编码 |

### 3. CSS Grid HTML 画布 🟥 自研

| 维度 | 评估 |
|------|------|
| **版本** | v6 (自研, `canvas-demo.html`) |
| **开源协议** | 私有 |
| **用途** | L4 画布 UI 的可视化工作流编辑器 |
| **集成深度** | 全自研 — 纯 HTML/CSS/JavaScript，无第三方 UI 框架依赖 |
| **替代方案** | React Flow、DeltaXML、D3.js 工作流编辑器 |
| **选型理由** | ① AgentFlow 的工作流展示是横平竖直的网格布局，不需要自由拖拽/连线 ② CSS Grid 天然支持此布局，零外部依赖 ③ 减少前端构建工具链（Webpack/Vite），直接作为静态文件提供服务 ④ 页面加载速度极快，无 React 运行时开销 |
| **画布结构** | 每个 Agent 节点在 CSS Grid 中占 4 列：步骤号 | Agent 卡片 | 箭头 | 输出卡片 |
| **技术栈** | 纯 HTML5 + CSS3 + ES6，通过 `fetch()` 调用后端 API，无任何 npm 依赖 |
| **风险** | ① 复杂的交互式编辑（拖拽/连线）不如 React Flow 方便 ② 大量节点时 CSS Grid 性能可能下降 |
| **缓解措施** | ① 当前设计为一键生成模式，不需要手动拖拽编辑，未来如需编辑可引入 React Flow ② 限制单次工作流节点数在 20 以内 |

### 4. GLM-5-Turbo API 🔗 集成

| 维度 | 评估 |
|------|------|
| **版本** | GLM-5-Turbo (智谱 AI) |
| **开源协议** | 商业 API（通过 HTTP 调用） |
| **用途** | Stage 0 源编排的核心 LLM 引擎 — 将用户一句话需求拆解为 N 个 Agent 节点 |
| **集成深度** | 浅 — 通过标准 OpenAI 兼容 HTTP API 调用 |
| **替代方案** | DeepSeek API、GPT-4o API、Qwen API |
| **选型理由** | ① GLM-5-Turbo 在中文理解和任务拆解方面表现优异 ② 智谱 API 完全兼容 OpenAI 格式，切换成本极低 ③ 作为源编排专用模型，不占用主执行模型的 Token 配额 |
| **风险** | ① API 可用性和延迟 ② Token 成本 ③ 模型能力变更 |
| **缓解措施** | ① 内置本地模板降级机制，API 不可用时自动切换到预置模板 ② 缓存策略：对相同需求命中缓存 ③ 可通过环境变量 `ZAI_API_KEY` 切换为其他 OpenAI 兼容 API |

**集成模式：**

```
┌──────────────────────────────────────────┐
│             AgentFlow 系统                │
│                                           │
│  /api/decompose ──HTTP──► GLM-5-Turbo    │
│                        API               │
│                                           │
│  所有交互通过标准 HTTP POST + JSON        │
│  不依赖任何 SDK，使用 urllib 原生调用     │
│  切换供应商只需修改 model + api_key       │
└──────────────────────────────────────────┘
```

**降级模板机制：**

当 GLM-5-Turbo API 不可用时，系统自动使用内置本地模板：

```python
# 根据需求关键词匹配模板
if "pid" in text or "控制器" in text:
    return PID_CONTROLLER_TEMPLATE  # 6节点
elif "网站" in text or "官网" in text:
    return WEBSITE_TEMPLATE         # 7节点
elif "训练" in text or "模型" in text:
    return ML_TEMPLATE              # 7节点
else:
    return GENERIC_TEMPLATE         # 5节点
```

### 5. validate_workflow() 自研 🟥

| 维度 | 评估 |
|------|------|
| **版本** | v1.0 (自研, `agentflow_schema.py`) |
| **开源协议** | 私有 |
| **用途** | Stage 4 的 WorkflowJSON 基本完整性校验 |
| **集成深度** | 全自研 — 约 20 行 Python 代码 |
| **替代方案** | JSON Schema (Ajv/jsonschema)、Pydantic |
| **选型理由** | ① 当前验证需求极简单（检查节点引用、profile 合法性），不需要 JSON Schema 的重量级方案 ② 减少外部依赖 ③ 未来如需复杂 Schema 验证可随时切换到 jsonschema |
| **风险** | 极低 — 功能简单，易于维护 |
| **未来升级路径** | 如需要更复杂的 Schema 验证（嵌套结构、条件验证等），可引入 `jsonschema` 库或 Pydantic v2 |

## 自研决策树

AgentFlow 在组件选型时遵循以下决策树：

```
该功能是 AgentFlow 的核心竞争力？ ──Yes──→ 自研 🟥
       │
       No
       ▼
该功能仅需调用外部 LLM API？ ──Yes──→ 集成 🔗
       │
       No
       ▼
OSS 市场有成熟可信的解决方案？
且协议兼容、无需深度定制？ ──Yes──→ 复用 🟦 (目前无此类需求)
       │
       No
       ▼
自研 🟥
```

## 长期维护策略

1. **环境变量配置** — 所有 API Key 和 URL 通过环境变量注入，不硬编码在代码中，支持随时切换供应商
2. **降级机制** — 每个 LLM API 调用都有本地降级方案，确保 API 不可用时系统仍可运行
3. **模型无关设计** — Compiler 生成的 Prompt 与具体模型无关，可通过 `AGENT_MODEL` 环境变量全局切换执行模型
4. **无外部依赖** — 核心代码仅依赖 Python 标准库（无需 `pip install`），降低部署和版本管理复杂度
5. **渐进增强** — 未来如需更复杂的能力（如 JSON Schema 验证、Redis 任务队列），可逐步引入，不影响现有架构
