# 06.2 — 安全架构

Agent 运行时采用 **纵深防御** 策略，从密钥管理、容器隔离、资源控制、审计追踪四个维度构建安全屏障。所有设计遵循最小权限和默认安全原则。

---

## 1. 密钥管理 (KMS)

### 架构

```
  ┌─────────────┐      ┌──────────────┐      ┌─────────────┐
  │   Agent     │      │  KMS 服务    │      │  外部 API   │
  │  (容器内)   │─────▶│  (运行时内)  │─────▶│   (OpenAI   │
  │             │      │              │      │   等)       │
  └─────────────┘      └──────────────┘      └─────────────┘
       │                      │                     │
       │  请求 Session Token  │                     │
       │◀─────── 返回 Token   │                     │
       │                      │  携带 Token 调用    │
       │                      │◀────────────────────│
       │                      │  验证 Token 签名    │
       │                      │────────────────────▶│
```

### 核心原则

- **运行时不持有原始 API Key** — 原始密钥仅存储在 KMS 的加密存储中，运行时进程不访问原始凭证
- **限时 Session Token** — KMS 为每个 Agent 实例签发带有过期时间的 Session Token（默认 TTL = 任务超时 + 30s 缓冲）
- **Scope 约束** — Token 绑定到具体的 Agent 实例 ID，无法被其他实例复用
- **审计记录** — 每次 Token 签发、验证和吊销均写入审计日志

### Token 格式

```json
{
  "iss": "agentflow-kms",
  "sub": "agent-instance-uuid-xxxxx",
  "aud": "api.openai.com",
  "exp": 1718000000,
  "iat": 1717999700,
  "scope": {
    "agent_type": "software-dev",
    "allowed_models": ["gpt-4", "gpt-4-turbo"],
    "max_tokens": 4096
  },
  "rate_limit": {
    "rpm": 60,
    "tpm": 100000
  }
}
```

## 2. 容器安全

每个 Agent 实例在独立容器中执行，容器配置如下：

### 安全配置基线

```yaml
container_security:
  # 禁用所有 Linux Capability
  cap_drop: ALL

  # 根文件系统只读（防止恶意写入）
  read_only_rootfs: true

  # 网络隔离
  network: none

  # 资源硬限制
  memory: "1G"
  cpus: 2
  pids_limit: 200

  # 挂载：仅挂载需要的卷
  volumes:
    - type: tmpfs
      target: /tmp
      options: ["noexec", "nosuid", "nodev", "size=100M"]
    - type: volume
      source: agent-output-<instance_id>
      target: /output

  # 进程隔离
  security_opt:
    - no-new-privileges:true
    - seccomp:default
    - apparmor:agentflow-profile
```

### 配置解读

| 配置项 | 目的 |
|--------|------|
| `--cap-drop=ALL` | 移除所有内核能力，容器进程以非特权用户运行，无法执行敏感系统调用 |
| `--read-only-root-filesystem` | 容器文件系统只读，防止植入恶意二进制文件或修改系统配置 |
| `--network none` | 禁用网络，Agent 仅能通过运行时代理（sidecar）与外界通信，所有流量受审计 |
| `--memory=1G --cpus=2` | 限制内存和 CPU 使用，防止资源耗尽攻击 |
| `--pids-limit=200` | 限制进程数量，防止 fork 炸弹 |
| 独立卷 | 每个 Agent 实例拥有独立的临时卷，实例间数据不互通 |
| `no-new-privileges` | 禁止进程提权 |
| seccomp / AppArmor | 内核级系统调用过滤器，仅允许 Agent 必要的最小系统调用集合 |

### 通信代理

由于容器网络被禁用，Agent 通过 Unix Domain Socket 与运行时的 sidecar 进程通信，由 sidecar 转发外部 API 请求：

```
┌─────────────────────┐      Unix Socket      ┌──────────────────┐
│  Agent 容器         │◄─────────────────────▶│  Sidecar 进程    │
│  (--network none)   │                        │  (运行时主机)    │
│                     │  请求转发 + Token 注入  │                  │
└─────────────────────┘                        └──────┬───────────┘
                                                       │ HTTPS
                                                       ▼
                                                ┌──────────────────┐
                                                │  外部 API        │
                                                │  (OpenAI 等)     │
                                                └──────────────────┘
```

## 3. 风险矩阵

| 风险 | 等级 | 描述 | 缓解措施 |
|------|------|------|----------|
| **Docker 逃逸** | 🔴 严重 | Agent 突破容器隔离，访问宿主机或其它容器 | `--cap-drop=ALL`, seccomp, AppArmor, 只读根文件系统, no-new-privileges |
| **密钥泄露** | 🔴 严重 | 原始 API Key 被 Agent 窃取 | 运行时不持有原始 Key；Session Token 限时、限 scope、审计追踪 |
| **Orchestrator 单点** | 🔴 严重 | Orchestrator 进程被攻破可控制所有 Agent | Orchestrator 以非 root 运行，各 Agent 实例独立隔离，最小化 Orchestrator 攻击面 |
| **共享卷攻击** | 🟠 高 | Agent 通过共享卷读取或污染其它 Agent 的数据 | 每个 Agent 使用独立卷；卷在 Agent 清理后立即销毁 |
| **用量炸弹** | 🟠 高 | Agent 发起大量 API 调用导致配额耗尽或费用激增 | 资源硬限制（内存/CPU/PID）、超时控制、Token 级别 rate limit、费用告警 |
| **输出泄露** | 🟠 高 | Agent 输出包含敏感信息，被非授权方读取 | 输出卷权限 0700，仅 Orchestrator 和授权用户可读取 |

### 风险应对策略

1. **预防优先** — 通过配置基线在源头消除大部分攻击面
2. **检测补充** — 运行时行为监控 + 异常检测
3. **隔离兜底** — 即使检测失败，容器隔离限制爆炸半径
4. **审计追溯** — 所有事件可审计，支持事后取证

## 4. 审计日志

### 设计原则

- **Append-Only** — 日志只能追加，不可删除或修改（通过文件系统权限 + 审计守护进程保障）
- **结构化** — 每条日志为 JSON 格式，包含事件类型、时间戳、Agent ID、操作详情
- **不可抵赖** — 日志条目使用 HMAC 链式签名，篡改可检测

### 日志条目格式

```json
{
  "id": "audit-001234",
  "timestamp": "2026-06-09T22:23:00.000Z",
  "event": "agent.created",
  "agent_id": "agent-xxxxxxxx",
  "agent_type": "software-dev",
  "session_id": "session-yyyyyyyy",
  "actor": "user-zzzzzzzz",
  "details": {
    "type_id": "software-dev",
    "timeout_s": 300,
    "retry": 1
  },
  "hmac": "a1b2c3d4e5..."
}
```

### 审计事件分类

| 事件类型 | 说明 |
|----------|------|
| `agent.created` | Agent 实例创建 |
| `agent.scheduled` | Agent 被调度到执行节点 |
| `agent.started` | 容器启动，开始执行 |
| `agent.completed` | 执行成功完成 |
| `agent.failed` | 执行失败（含失败原因） |
| `agent.cleaned` | 容器销毁，资源回收 |
| `agent.timeout` | 超时强制终止 |
| `agent.retry` | 自动重试 |
| `token.issued` | KMS 签发 Session Token |
| `token.revoked` | Token 被吊销 |
| `token.validation_failed` | Token 验证失败（签名/过期/scope 不匹配） |
| `security.violation` | 安全策略违规（如资源超限、越权访问） |

### 审计日志存储

```yaml
audit_log:
  storage: "file"                    # file | s3 | elasticsearch
  path: "/var/log/agentflow/audit/"
  rotation:
    max_size_mb: 500
    max_age_days: 90
    compress: true
  append_only:
    enabled: true
    guard_process: "auditd-agentflow"  # 系统级审计守护进程
    permissions: "0400"               # 仅 root 可读取，无人可写入（由守护进程写入）
```

### HMAC 链式签名

审计日志的完整性通过 HMAC 链保证：

```
HMAC_0 = HMAC_SHA256("seed", log_entry_0)
HMAC_1 = HMAC_SHA256(HMAC_0, log_entry_1)
HMAC_2 = HMAC_SHA256(HMAC_1, log_entry_2)
...
```

每条日志的 HMAC 依赖上一条日志的 HMAC，形成签名链。修改任意一条日志会导致链断裂，可被检测工具发现。
