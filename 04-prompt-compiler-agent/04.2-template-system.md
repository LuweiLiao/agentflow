     1|# 04.2 — JSON 模板系统设计
     2|
     3|## 概述
     4|
     5|AgentFlow 的模板系统是 Compiler 的"知识库"。每个模板定义了一种 Agent 角色的行为模式——包括 system prompt、工具集、超时配置和提示词模板。
     6|
     7|模板以 **JSON 格式** 存储在 `04.3-templates/` 目录下。Compiler 的 `TemplateEngine` 在编译时根据节点的 `profile` 字段加载对应模板进行渲染。
     8|
     9|```
    10|节点 profile = "analysis"
    11|    │
    12|    ▼
    13|TemplateEngine.load("analysis")
    14|    │  加载 04.3-templates/analysis.json
    15|    │
    16|    ▼
    17|模板 dict:
    18|  ├── system_prompt    → Agent 角色定义
    19|  ├── tool_set         → 可用工具列表
    20|  ├── max_turns        → 最大交互轮数
    21|  ├── timeout_s        → 超时秒数
    22|  ├── output_format    → 输出格式说明
    23|  └── prompt_template  → 提示词模板（含 {变量占位符}）
    24|    │
    25|    ▼  + 变量渲染
    26|PromptTask.prompt (完整的自然语言提示词)
    27|```
    28|
    29|## 模板结构
    30|
    31|每个 JSON 模板包含以下字段：
    32|
    33|```yaml
    34|# AgentFlow 模板 — analysis.json
    35|# 分析型 Agent：需求分析、数据调研、问题定义
    36|profile: analysis            # 模板标识，与 NodeDef.profile 匹配
    37|label: 分析型                # 人类可读的名称
    38|icon: 🔍                    # 图标
    39|
    40|system_prompt: |            # Agent 的 system prompt（角色定义）
    41|  你是 AgentFlow 工作流中的分析型 Agent（analysis）。
    42|  你的职责是：分析需求、调研数据、定义问题。
    43|  你的思考必须是彻底和全面的。
    44|
    45|tool_set:                   # 可用工具列表
    46|  - bash
    47|  - read
    48|  - write
    49|
    50|max_turns: 10               # 最大 Think→Act→Observe 轮数
    51|timeout_s: 120              # 执行超时（秒）
    52|
    53|output_format: |            # 输出格式说明（嵌入到 prompt 中）
    54|  ```json
    55|  {
    56|    "summary": "一句话总结",
    57|    "output": "详细分析报告",
    58|    "files": ["生成的文件列表"],
    59|    "metrics": { ... }
    60|  }
    61|  ```
    62|
    63|prompt_template: |          # 提示词模板（含变量占位符）
    64|  你是一个 {node.label} Agent（analysis 角色）。
    65|
    66|  ## 全局上下文
    67|  {global.goal}
    68|
    69|  {global.constraints}
    70|
    71|  ## 本节点任务
    72|  {params.description}
    73|
    74|  ## 上游输入
    75|  {upstream_context}
    76|
    77|  ## 要求
    78|  1. 对任务进行深入分析，拆解关键问题
    79|  2. 确定影响范围和技术约束
    80|  3. 输出结构化分析结果
    81|  4. 使用中文回复
    82|```

    95|## 变量渲染机制
    96|
    97|### 支持变量
    98|
    99|| 变量 | 来源 | 示例值 |
   100||------|------|--------|
   101|| `{global.goal}` | `workflow.global_context["goal"]` | 用户输入的 requirement |
   102|| `{global.constraints}` | `workflow.global_context["constraints"]` | 列表→`- 约束1\n- 约束2` |
   103|| `{node.label}` | `node.label` | "需求分析" |
   104|| `{params.description}` | `node.desc` | "解析用户需求" |
   105|| `{params.*}` | `node.params.*` | 自定义参数 |
   106|| `{upstream_context}` | 上游节点结果汇聚 | "来自[需求分析]的输出：..." |
   107|| `{sequence}` | 执行序号 | "1", "2", "3" |
   108|
   109|### 渲染规则
   110|
   111|1. **全局变量**（`global.`）：从 `workflow.global_context` 读取
   112|2. **节点变量**（`node.`）：从 `NodeDef` 的字段读取（label, desc, id）
   113|3. **参数变量**（`params.` / `param.`）：从 `node.params` 或 `node.desc` 读取
   114|4. **上游上下文**（`upstream_context`）：编译时从 DAG 边收集上游节点信息
   115|5. **列表类型**：自动转换为多行 `- 项` 格式
   116|6. **变量不存在**：保留原始 `{key}` 样式，不报错
   117|
   118|### 渲染实现
   119|
   120|```python
   121|# TemplateEngine.render() 使用正则替换
   122|re.sub(r"\{([a-z_.]+)\}", replace_var, template_str)
   123|```

   125|## 模板加载机制
   126|
   127|```python
   128|class TemplateEngine:
   129|    def load(self, profile: str) -> dict:
   130|        # 1. 检查缓存
   131|        if profile in self._cache:
   132|            return self._cache[profile]
   133|
   134|        # 2. 加载 JSON 文件
   135|        path = os.path.join(self.template_dir, f"{profile}.json")
   136|
   137|        # 3. 文件不存在 → 回退到 dev.json
   138|        if not os.path.isfile(path):
   139|            fallback = os.path.join(self.template_dir, "dev.json")
   140|            path = fallback
   141|
   142|        # 4. 解析 JSON
   143|        with open(path) as f:
   144|            tmpl = yaml.safe_load(f)
   145|
   146|        # 5. 写入缓存
   147|        self._cache[profile] = tmpl
   148|        return tmpl
   149|```
   150|
   151|**降级策略：** 如果 `analysis.json` 不存在，自动回退到 `dev.json`。这确保了即使只有少量模板也能正常工作。

   153|## AgentRunner 如何使用 PromptTask
   154|
   155|下游的 `AgentRunner.execute()` 接收到 `PromptTask` 后：
   156|
   157|1. 设置 `system_prompt` 为模板中的 system prompt
   158|2. 使用 `prompt` 作为用户消息
   159|3. 在 `work_dir` 中执行 Think→Act→Observe 循环
   160|4. 使用 `tool_set` 限制可用工具
   161|5. 受 `max_turns` 和 `timeout_s` 约束
   162|
   163|```
   164|AgentRunner.execute(prompt, system_prompt, tools, max_turns, timeout)
   165|    │
   166|    ▼
   167|┌─────────────────────────────────────────┐
   168|│  Agent Loop                              │
   169|│                                           │
   170|│  Think → Act (call tool) → Observe       │
   171|│                                           │
   172|│  工具: bash, read, write (从 tool_set)   │
   173|│  最大轮数: max_turns                      │
   174|│  超时: timeout_s                          │
   175|└─────────────────────────────────────────┘
   176|    │
   177|    ▼
   178|EnvelopeJSON (结果封装)
   179|```

   181|## 模板设计原则
   182|
   183|### 1. 角色单一职责
   184|每个模板对应一种 Agent 角色。分析型只做分析，开发型只做开发，不混入多种职责。
   185|
   186|### 2. 参数化
   187|所有可变内容通过 `{params.*}` 和 `{global.*}` 注入。模板本身保持通用和不变。
   188|
   189|### 3. 上游感知
   190|通过 `{upstream_context}` 获取上游节点的输出，实现链式推理和上下文传递。
   191|
   192|### 4. 工具约束
   193|`tool_set` 限制 Agent 可用的工具。分析型不需要部署能力，开发型不需要输出报告。
   194|
   195|### 5. 渐进式复杂度
   196|简单模板（如 analysis 的 10 轮）可作为复杂工作流的基础步骤。需要更多迭代的节点（如 deploy 的 300s）配置更宽松的超时。
   197|
   198|### 6. 降级安全
   199|模板不存在时自动回退到 `dev.json`，保证工作流不会因为缺少模板而中断。
