# 04.2 — 模板系统设计

## 概述

AgentFlow 的模板系统是 Compiler Agent 的 "知识库"。每个模板定义了一种工作流类型，描述了该类型工作流的节点结构、参数映射和上下游数据流。

模板以 YAML 格式存储，位于 `04.3-templates/` 目录下。Compiler Agent 在编译时根据 WorkflowJSON 中每个节点的 `template` 字段加载对应模板进行渲染。

## 模板与 WorkflowJSON 的关系

```
WorkflowJSON (声明式)
    │
    │  nodes[].template = "algorithm-designer"
    │  nodes[].params = { ... }
    │
    ▼
Compiler Agent
    │
    │  加载 algorithm-designer.yaml
    │  执行变量渲染
    │
    ▼
PromptTask (具象化)
```

- **WorkflowJSON** 描述 "做什么"（声明式，高层抽象）
- **模板** 描述 "怎么说"（固定 Prompt 骨架 + 变量槽位）
- **PromptTask** 是最终产物（可执行的完整 Prompt）

## 模板结构

每个 YAML 模板包含以下顶级字段：

```yaml
# 模板元数据
name: string                    # 模板名称，与节点 template 字段匹配
version: string                 # 语义版本号
description: string             # 模板用途说明

# 模板引擎配置
engine:
  runtime: "claude-code"        # 目标运行时
  model_recommendation: string  # 推荐的模型（如 claude-sonnet-4, deepseek-v4）

# Prompt 模板
prompt_template: string         # Jinja2 风格的 Prompt 模板字符串
                                # 支持 {global.goal}, {params.xxx}, {output_schema},
                                # {upstream_node_label}, {upstream_output},
                                # {timeout_s}, {retry}, {global.constraints}

# 输出 Schema 定义
output_schema:
  type: object
  properties: { ... }           # 节点输出的 JSON Schema

# 参数定义（可选）
params_schema:
  type: object
  properties: { ... }           # 节点 params 的 JSON Schema，用于校验输入参数
  required: [ ... ]

# 默认值（可选）
defaults:
  timeout_s: 300                # 默认超时
  retry: 3                      # 默认重试次数
```

## 变量渲染机制

### 内置变量

| 变量 | 来源 | 示例值 |
|------|------|--------|
| `{global.goal}` | `global_context.goal` | "设计一个高性能排序算法" |
| `{params.xxx}` | `node.params.xxx` | `{params.language}` → "Python" |
| `{output_schema}` | `node.output_schema` (序列化) | `{"type":"object",...}` |
| `{upstream_node_label}` | 上游节点 `label` 拼接 | "需求分析, 架构设计" |
| `{upstream_output}` | 上游输出样例 | "用户需要..." 或实际 JSON |
| `{timeout_s}` | `node.timeout_s` | `300` |
| `{retry}` | `node.retry` | `3` |
| `{global.constraints}` | `global_context.constraints[]` | 带 `- ` 前缀的多行文本 |

### 渲染管道

```
                            ┌──────────────────┐
                            │   WorkflowJSON    │
                            │  global_context   │
                            │  nodes[].params   │
                            └────────┬─────────┘
                                     │
                                     ▼
┌─────────────────────────────────────────────────────┐
│               Template Resolver                      │
│                                                       │
│  nodes[].template ────► 加载对应的 .yaml 模板         │
│                                                       │
│  prompt_template  +  变量映射  ────►  渲染 Prompt     │
│                                                       │
│  输出：PromptTask.prompt                              │
└─────────────────────────────────────────────────────┘
```

## 模板设计原则

### 1. 单一职责
每个模板对应一种工作流模式。不要将多种不相关的工作流逻辑混入同一模板。

### 2. 参数化
所有可变内容应通过 `{params.xxx}` 注入，避免硬编码。模板本身应保持通用。

### 3. 上游感知
通过 `{upstream_node_label}` 和 `{upstream_output}` 感知上游上下文，实现链式推理。

### 4. Schema 驱动
每个模板定义明确的输出 Schema，使下游节点可以预期上游输出的结构。

### 5. 渐进式复杂度
简单模板（如单节点）可作为复合模板的子步骤被组合调用。

## 模板注册流程

```
1. 在 04.3-templates/ 下创建 .yaml 文件
2. 定义 name、prompt_template、output_schema
3. 在 WorkflowJSON 的节点中引用 template = <name>
4. Compiler Agent 自动加载并渲染
```

无需手动注册 — Compiler Agent 在编译时扫描 `04.3-templates/` 目录下的所有 `.yaml` 文件，按 `name` 字段匹配。

## 最佳实践

- **模板版本管理**：使用 `version` 字段跟踪变更，编译器可选择是否兼容旧版本
- **参数校验**：`params_schema` 帮助早发现参数缺失或类型错误
- **默认值**：`defaults` 减少重复配置，同时允许 WorkflowJSON 覆盖
- **注释**：YAML 支持 `#` 注释，在模板中解释关键设计决策
- **测试**：每个模板应有对应的 WorkflowJSON 测试用例，确保渲染结果符合预期
