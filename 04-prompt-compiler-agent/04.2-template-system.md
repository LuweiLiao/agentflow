# 04.2 — YAML 模板系统设计

## 概述

AgentFlow 的模板系统是 Compiler 的"知识库"。每个模板定义了一种 Agent 角色的行为模式——包括 system prompt、工具集、超时配置和提示词模板。

模板以 **YAML 格式** 存储在 `04.3-templates/` 目录下。Compiler 的 `TemplateEngine` 在编译时根据节点的 `profile` 字段加载对应模板进行渲染。

```
节点 profile = "analysis"
    │
    ▼
TemplateEngine.load("analysis")
    │  加载 04.3-templates/analysis.yaml
    │
    ▼
模板 dict:
  ├── system_prompt    → Agent 角色定义
  ├── tool_set         → 可用工具列表
  ├── max_turns        → 最大交互轮数
  ├── timeout_s        → 超时秒数
  ├── output_format    → 输出格式说明
  └── prompt_template  → 提示词模板（含 {变量占位符}）
    │
    ▼  + 变量渲染
PromptTask.prompt (完整的自然语言提示词)
```

## 模板结构

每个 YAML 模板包含以下字段：

```yaml
# AgentFlow 模板 — analysis.yaml
# 分析型 Agent：需求分析、数据调研、问题定义
profile: analysis            # 模板标识，与 NodeDef.profile 匹配
label: 分析型                # 人类可读的名称
icon: 🔍                    # 图标

system_prompt: |            # Agent 的 system prompt（角色定义）
  你是 AgentFlow 工作流中的分析型 Agent（analysis）。
  你的职责是：分析需求、调研数据、定义问题。
  你的思考必须是彻底和全面的。

tool_set:                   # 可用工具列表
  - bash
  - read
  - write

max_turns: 10               # 最大 Think→Act→Observe 轮数
timeout_s: 120              # 执行超时（秒）

output_format: |            # 输出格式说明（嵌入到 prompt 中）
  ```json
  {
    "summary": "一句话总结",
    "output": "详细分析报告",
    "files": ["生成的文件列表"],
    "metrics": { ... }
  }
  ```

prompt_template: |          # 提示词模板（含变量占位符）
  你是一个 {node.label} Agent（analysis 角色）。

  ## 全局上下文
  {global.goal}

  {global.constraints}

  ## 本节点任务
  {params.description}

  ## 上游输入
  {upstream_context}

  ## 要求
  1. 对任务进行深入分析，拆解关键问题
  2. 确定影响范围和技术约束
  3. 输出结构化分析结果
  4. 使用中文回复
```

## 变量渲染机制

### 支持变量

| 变量 | 来源 | 示例值 |
|------|------|--------|
| `{global.goal}` | `workflow.global_context["goal"]` | 用户输入的 requirement |
| `{global.constraints}` | `workflow.global_context["constraints"]` | 列表→`- 约束1\n- 约束2` |
| `{node.label}` | `node.label` | "需求分析" |
| `{params.description}` | `node.desc` | "解析用户需求" |
| `{params.*}` | `node.params.*` | 自定义参数 |
| `{upstream_context}` | 上游节点结果汇聚 | "来自[需求分析]的输出：..." |
| `{sequence}` | 执行序号 | "1", "2", "3" |

### 渲染规则

1. **全局变量**（`global.`）：从 `workflow.global_context` 读取
2. **节点变量**（`node.`）：从 `NodeDef` 的字段读取（label, desc, id）
3. **参数变量**（`params.` / `param.`）：从 `node.params` 或 `node.desc` 读取
4. **上游上下文**（`upstream_context`）：编译时从 DAG 边收集上游节点信息
5. **列表类型**：自动转换为多行 `- 项` 格式
6. **变量不存在**：保留原始 `{key}` 样式，不报错

## 模板加载机制

```python
class TemplateEngine:
    def load(self, profile: str) -> dict:
        # 1. 检查缓存
        if profile in self._cache:
            return self._cache[profile]

        # 2. 加载 YAML 文件
        path = os.path.join(self.template_dir, f"{profile}.yaml")

        # 3. 文件不存在 → 回退到 dev.yaml
        if not os.path.isfile(path):
            fallback = os.path.join(self.template_dir, "dev.yaml")
            path = fallback

        # 4. 解析 YAML
        with open(path) as f:
            tmpl = yaml.safe_load(f)

        # 5. 写入缓存
        self._cache[profile] = tmpl
        return tmpl
```

**降级策略：** 如果 `analysis.yaml` 不存在，自动回退到 `dev.yaml`。这确保了即使只有少量模板也能正常工作。

## AgentRunner 如何使用 PromptTask

下游的 `AgentRunner.execute()` 接收到 `PromptTask` 后：

1. 设置 `system_prompt` 为模板中的 system prompt
2. 使用 `prompt` 作为用户消息
3. 在 `work_dir` 中执行 Think→Act→Observe 循环
4. 使用 `tool_set` 限制可用工具
5. 受 `max_turns` 和 `timeout_s` 约束

```
AgentRunner.execute(prompt, system_prompt, tools, max_turns, timeout)
    │
    ▼
┌─────────────────────────────────────────┐
│  Agent Loop                              │
│                                           │
│  Think → Act (call tool) → Observe       │
│                                           │
│  工具: bash, read, write (从 tool_set)   │
│  最大轮数: max_turns                      │
│  超时: timeout_s                          │
└─────────────────────────────────────────┘
    │
    ▼
EnvelopeJSON (结果封装)
```

## 模板设计原则

### 1. 角色单一职责
每个模板对应一种 Agent 角色。分析型只做分析，开发型只做开发，不混入多种职责。

### 2. 参数化
所有可变内容通过 `{params.*}` 和 `{global.*}` 注入。模板本身保持通用和不变。

### 3. 上游感知
通过 `{upstream_context}` 获取上游节点的输出，实现链式推理和上下文传递。

### 4. 工具约束
`tool_set` 限制 Agent 可用的工具。分析型不需要部署能力，开发型不需要输出报告。

### 5. 渐进式复杂度
简单模板（如 analysis 的 10 轮）可作为复杂工作流的基础步骤。需要更多迭代的节点（如 deploy 的 300s）配置更宽松的超时。

### 6. 降级安全
模板不存在时自动回退到 `dev.yaml`，保证工作流不会因为缺少模板而中断。
