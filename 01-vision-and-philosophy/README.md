# AgentFlow — 愿景与设计哲学

> **用户说需求 → AI 画草稿 → 用户完善 → 自动执行 → 结果反馈**

---

## 1. 核心动机：为什么做 AgentFlow？

现有工作流平台（n8n、Airflow、Dify）都建立在同一个前提之上：**用户必须精确地描述每一步该做什么**。无论是拖拽节点、编写 DAG 还是配置 Prompt Template，编排权始终在用户手里，AI 只是执行工具。

AgentFlow 的出发点是**颠倒这个关系**。

我们观察到两个趋势正在交汇：

- **LLM 已经足够聪明**，能够理解模糊、高层次的用户需求，并自行拆解为可执行的步骤序列。
- **执行环境已经足够成熟**，Docker 容器可以安全、隔离地运行任意代码，Claude Code / Codex 等 Agent 可以在沙箱中自主完成复杂任务。

这意味着，从"用户手动编排"到"用户表达意图，AI 自动编排"的范式转移已经具备技术条件。

**AgentFlow 的核心动机很简单**：让 AI 成为工作流的构造者，而非仅仅是执行者。用户从"编程式编排"中解放出来，回归到"需求表达者"和"结果评判者"的角色。

## 2. 设计哲学

AgentFlow 的设计围绕三个核心信念展开。

### 2.1 草稿优先，迭代完善

> AI 的第一版永远不完美，但它是最好的沟通起点。

用户不需要一次性给出完整的工作流定义。AgentFlow 接受模糊、零散、甚至自相矛盾的自然语言需求，AI 据此生成可视化草稿。用户在草稿上圈点、修改、补充，AI 据此调整。这个过程是**协作式的**，而非单向的配置。

这与传统平台的差异显著：

| 方面 | n8n / Airflow / Dify | AgentFlow |
|------|----------------------|-----------|
| 入门门槛 | 需要理解节点、触发器、DAG 等概念 | 说一句话即可开始 |
| 修改方式 | 手动拖拽 / 编辑代码 | 自然语言反馈 + AI 自动调整 |
| 知识要求 | 熟悉平台语义和集成细节 | 只需懂自己的业务需求 |

### 2.2 画布即协议

AgentFlow 的视觉画布不是最终产物，而是**用户与 Compiler Agent 之间的共享协议**。

- 画布上的每个节点代表一个**目标**而非一条**指令**。
- 节点之间的连线代表**信息依赖**而非**执行顺序**。
- 画布的整体布局反映了用户对问题结构的理解。

这个协议是双向的：用户通过画布向 AI 传达自己的思维方式，AI 通过画布向用户展示自己的理解和规划方案。

### 2.3 编译器模式，而非模板模式

Dify 使用 Prompt Template 来定义 Agent 行为，n8n 使用预构建的集成节点。AgentFlow 走了一条不同的路：

每个画布节点都经过 **Compiler Agent** 编译为**完整的自然语言提示词**。这个提示词包含了节点目标、上下文、前置节点的输出摘要、全局约束等所有信息。编译后的提示词被传递给 Claude Code（或其他执行 Agent），在隔离的 Docker 容器中执行。

这意味着：
- Agent 的行为由**动态生成的提示词**驱动，而非静态模板。
- 每个节点都可以根据前置结果自适应调整自己的执行策略。
- 不需要为每个新场景编写新模板，Compiler Agent 会自动适配。

## 3. 核心差异化：源编排（Source Orchestration）

AgentFlow 最核心的概念是 **源编排（Source Orchestration）**，这是它与所有现有编排平台最根本的区别。

### 3.1 定义

**源编排**是指：用户编排的不是任务的执行流程，而是**信息的来源、约束和期望的输出形态**。执行路径本身由 AI 在运行时决定。

传统编排是**命令式的**：

```
步骤1: 调用API获取数据
步骤2: 清洗数据
步骤3: 调用LLM分析
步骤4: 发送报告
```

源编排是**声明式的**：

```
数据源: 销售数据库 (schema X)
约束: 只分析最近30天
产出: 一份包含趋势分析和Top 5建议的报告
```

AI 自动决定是否需要先写 SQL 查询，用什么方法清洗，调用哪个模型分析，以及如何组织报告格式。

### 3.2 为什么这是范式转移？

| 维度 | 任务编排（Task Orchestration） | 源编排（Source Orchestration） |
|------|-------------------------------|-------------------------------|
| 用户描述 | 怎么做 | 要什么 |
| 变更成本 | 修改 DAG / 重新拖拽 | 修改约束描述，AI 重新规划 |
| 可扩展性 | 每增加一个场景需要手动配置 | 场景自动泛化 |
| 错误恢复 | 重试/死信队列 | 重新规划路径 |
| 人机角色 | 人是程序员，机器是执行者 | 人是需求方，机器是规划+执行者 |

### 3.3 Compiler Agent 的角色

Compiler Agent 是源编排的核心引擎。它做的事情可以概括为：

1. **接收** — 解析画布结构、节点属性、用户标注
2. **推理** — 理解每个节点在整个工作流中的角色、前置依赖、产出要求
3. **编译** — 为每个节点生成包含完整上下文的提示词，确保执行 Agent 不需要跨节点通信就能独立完成任务
4. **反馈** — 收集执行结果，更新画布状态，标识异常和偏差

这本质上是一个**从声明式描述到可执行计划**的编译过程，类似高级语言到机器码的编译，但这里的"机器码"是自然语言提示词，"执行环境"是配备了 Claude Code 的 Docker 容器。

## 4. 与控制理论的联系

AgentFlow 的设计深受控制理论的启发。这不是一个比喻，而是系统架构层面的对应。

### 4.1 闭环反馈架构

AgentFlow 将每个工作流视为一个**反馈控制系统**：

```
        期望产出         实际产出
用户需求 ──→ [Controller] ──→ [Plant (Agent)] ──→ 输出
                  ↑                        │
                  │       ┌───────────┐    │
                  └───────│ 反馈回路   │←───┘
                          └───────────┘
                          (执行结果评估)
```

- **Controller** = Compiler Agent（根据反馈调整提示词编译策略）
- **Plant** = 执行 Agent（Claude Code in Docker）
- **Reference** = 用户在画布上定义的期望产出
- **Output** = 执行结果
- **Feedback** = 执行日志、产出质量评估、异常报告

当执行结果偏离期望时，反馈回路驱动 Controller 调整编译策略（也许需要更详细的提示词、更严格的约束、或者重新拆分节点）。

### 4.2 Lyapunov 稳定

在控制理论中，Lyapunov 稳定性描述的是系统在受到扰动后是否能回到平衡状态。AgentFlow 用这个概念来设计工作流的**鲁棒性**：

- **平衡点** = 用户对产出满意
- **扰动** = 中间步骤出错、API 返回异常、LLM 幻觉
- **恢复机制** = Compiler Agent 的重新规划 + 执行 Agent 的重试/绕行策略

一个 Lyapunov 稳定（在此语境下）的工作流，意味着无论中间出现什么偏差，系统都能通过闭环反馈回到产出令用户满意的状态。**稳定性不是来自每一步的确定性，而是来自反馈回路收敛的能力。**

这与传统工作流平台的"容错"有本质区别：

| | 传统容错 | AgentFlow 稳定 |
|--|---------|---------------|
| 策略 | 重试固定次数 | 重新规划执行路径 |
| 范围 | 同一步骤内 | 可跨步骤调整 |
| 灵活性 | 零（只重试相同操作） | 高（可以选择不同方法） |
| 保证 | 最大重试次数后失败 | 持续优化直到收敛或用户确认 |

### 4.3 可观性与可控性

AgentFlow 借鉴了控制理论中的另外两个核心概念：

- **可观性（Observability）**：系统必须能够通过输出（执行日志、中间产物、Agent 推理轨迹）推断出内部状态（哪个节点出了问题、为什么偏离）。AgentFlow 要求每个执行 Agent 输出结构化的执行轨迹，使 Compiler Agent 和用户都能理解系统的内部状态。
- **可控性（Controllability）**：系统必须能够通过调整输入（提示词修改、约束添加、节点重排）来影响输出。AgentFlow 的画布提供了丰富的"旋钮"供用户和 Compiler Agent 调节。

## 5. 适用场景与不适用场景

### ✅ 适合 AgentFlow 的场景

| 场景 | 原因 |
|------|------|
| **数据报告 / 分析管道** | 输入是数据库/API，产出是报告，路径可以动态规划 |
| **内容生产流水线** | 调研 → 撰写 → 审核 → 发布，每一步需要上下文和自主判断 |
| **代码迁移 / 重构** | 需要理解旧代码、生成新代码、运行测试、根据错误调整 |
| **文档生成与维护** | 从代码/设计文档自动生成多格式文档 |
| **研究综述** | 检索文献 → 分析对比 → 撰写综述 → 格式化引用 |
| **自动化运维剧本** | 故障排查、日志分析、修复建议、执行修复 |
| **个性化教育内容生成** | 根据学习者水平动态调整内容和习题 |

共同特点：**路径不固定、结果可判定、需要上下文理解、用户有能力判断产出质量但不想管理每一步细节。**

### ❌ 不适合 AgentFlow 的场景

| 场景 | 原因 |
|------|------|
| **毫秒级实时系统** | Agent 的规划-执行-反馈循环有较高延迟 |
| **确定性的批量 ETL** | 如果用 SQL 就能清楚描述的数据管道，Airflow 更轻量可靠 |
| **简单 CRUD 自动化** | 用 n8n 的两个节点就能解决的事情，不需要动态规划 |
| **合规要求固定执行路径的场景** | 金融审计、医疗流程等需要每一步都可复现、不可动态调整 |
| **纯数据处理（无 AI 判断）** | 如果所有节点都是确定性计算，Agent 的灵活性是多余的 |
| **高吞吐、低成本的场景** | Docker + Claude Code 的开销远高于脚本执行 |

### 5.1 决策树

用这个简单的问题来判断 AgentFlow 是否适合你的场景：

1. **你的工作流路径在执行前是否完全确定？**
   - 是 → 考虑 Airflow / n8n
   - 否 → 继续
2. **你是否愿意让 AI 在运行时决定执行路径？**
   - 否 → 坚持手动编排
   - 是 → 继续
3. **你对产出质量有清晰的评判标准吗？**
   - 否 → AgentFlow 暂时帮不了你（需要先定义评判标准）
   - 是 → **AgentFlow 适合你 🎯**

## 6. 结语

AgentFlow 代表了一种新的范式：**不是让人去适应编排工具的语义，而是让工具去理解人的意图。**

源编排、闭环反馈、编译器模式——这些设计选择共同指向一个目标：让工作流编排从"编程行为"变成"对话行为"。用户不再是程序员，而是指挥家。AI 不再是工具，而是乐团。
