# Phase 3 — 生产增强 (Week 9-12)

> 目标：从 MVP 原型升级为可内部使用的生产系统，重点提升安全、可靠和可用性。

---

## 1. 目标

- ✅ KMS 集成管理所有敏感凭据（API Key、Token）
- ✅ 模板管理 UI — 支持模板的创建、编辑、版本管理
- ✅ 并行执行 — DAG 中无依赖的节点可同时运行
- ✅ 闭环反馈 — 执行结果可回流到后续节点
- ✅ 执行历史 — 记录每次执行的详情，支持重放

## 5. 验收标准

| # | 验收条件 | 判定方式 |
|---|---|---|
| 1 | KMS 存储的 API Key 可在工作流执行时安全注入 | 安全测试 + 日志审计 |
| 2 | 模板可创建、编辑、发布新版本、回滚旧版本 | 人工测试 |
| 3 | DAG 中无依赖的节点可并行执行 (性能 > 2x 提升) | 性能测试 |
| 4 | 并行执行 20 个节点无竞态错误 | 压力测试 |
| 5 | 闭环反馈：下游节点可引用上游节点的执行结果 | 集成测试 |
| 6 | 执行历史完整记录每次执行的时间、结果、日志 | 人工测试 |
| 7 | 执行重放可复现历史结果 | 集成测试 |
| 8 | 内测用户可完成 "注册 → 创建模板 → 搭建工作流 → 执行 → 查看历史" 全流程 | 用户测试 |
| 9 | 代码覆盖率 > 70% | CI 报告 |
| 10 | 无 P0 Bug，P1 Bug < 5 个 | Bug 追踪系统 |

## 6. 风险 & 应对

| 风险 | 应对 |
|---|---|
| KMS 集成复杂度高 | MVP 阶段保留环境变量兜底，KMS 失败时降级警告 |
| 并行执行竞态条件 | 引入 Actor 模型隔离状态，避免共享内存 |
| 内测用户反馈量过大 | 建立反馈分类标签体系 (Bug / 需求 / 疑问) |
| 数据库设计变更 (执行历史) | 使用迁移工具 (goose / alembic)，确保向前兼容 |
