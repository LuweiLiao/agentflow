总体架构概览

🟦 直接复用 OSS 🟩 封装适配 🟥 自研核心 🟪 专用 Agent 角色

AgentFlow 是一个将可视化工作流编排与AI Agent 自动执行结合的平台。用户通过画布拖拽配线定义工作流 DAG，系统自动编译为 Agent 可执行的提示词，在 Docker 容器中通过 Claude Code CLI 驱动执行，形成完整的闭环反馈系统。

🎯 核心理念

每个画布节点 → 由 Compiler Agent 编译为完整自然语言提示词 → Claude Code 在 Docker 中执行 → 结果通过 Envelope 协议传递。

🔁 闭环反馈

下游 Agent 测试失败 → 自动回退到上游重新生成 → 形成控制理论中的负反馈闭环。每次迭代 dV/dt < 0。

数据契约定义

系统三份核心数据结构的完整 schema。

3.1 WorkflowJSON — 画布输出

📤 生产者

画布 UI (React Flow)。用户拖拽配线完成后导出。

📥 消费者

Prompt Compiler Agent。读取后编译为 PromptTask。

{ // ── 工作流元信息 ── "version": "1.0.0", "workflow_id": "wf_01J...", "name": "PID控制器全流程", // ── 全局上下文（所有节点共享） ── "global_context": { "goal": "为四旋翼无人机设计并实现PID控制器", "language": "zh-CN", "constraints": ["C++实现", "基于ArduPilot框架", "增益Kp范围0~10"] }, // ── 节点列表 ── "nodes": [ { "id": "alg", "type": "algorithm-designer", // → templates/algorithm-designer.yaml "label": "算法设计", "params": { "plant": "quadcopter", "method": "Ziegler-Nichols" }, "output_schema": { "type": "object", "properties": { "Kp": { "type": "number", "min": 0, "max": 10 }, "Ki": { "type": "number" }, "Kd": { "type": "number" } }}, "timeout_s": 120, "retry": 2, "on_failure": "retry" }, { // ... dev, test 节点类似（详见GitHub仓库）} ], "edges": [ { "source": "alg", "target": "dev" }, { "source": "dev", "target": "test" } ] }
⚠️ Schema 引用机制 input_schema 使用 JSON Schema $ref 语法引用上游节点的 output_schema。画布连线时，如果 A 的 output 不匹配 B 的 input，禁止连线或显示红色警告。

3.2 PromptTask — Compiler Agent 输出

📤 生产者

Prompt Compiler Agent。用模板渲染 WorkflowJSON 后生成。

📥 消费者

Orchestrator。读取后按拓扑序派发 Docker 容器。

[ { "task_id": "pt_alg_01J...", "workflow_id": "wf_01J...", "node_id": "alg", "node_type": "algorithm-designer", "sequence": 1, "parallel_group": null, "depends_on": [], // ── 核心 payload：完整的自然语言提示词 ── "prompt": "## 目标\n为四旋翼无人机设计PID控制器\n→ 使用Ziegler-Nichols方法整定参数\n\n## 约束\n- C++实现\n- 基于ArduPilot框架\n- 输出必须符合以下schema: {Kp, Ki, Kd}\n- 限时120秒\n- 失败可重试2次\n\n## 输出格式\nEnvelope JSON\n\n## 成功标准\npayload 通过 schema 验证", "output_schema": { "type": "object", "properties": { ... } }, "timeout_s": 120, "retry": 2 }, { ... dev, test ... } ]
💡 PromptTask 是系统的"核心货币" 所有的复杂性都在 Compiler Agent 的模板渲染步骤中被消化为一段纯粹的自然语言提示词。Orchestrator 和 Agent 不需要理解 WorkflowJSON 的结构，只需理解这段提示词。

3.3 Envelope JSON — Agent stdout 输出

📤 生产者

Claude Code Agent（Docker 容器内）。写入 stdout 后退出。

📥 消费者

Orchestrator。读取 stdout、校验 schema、传递到下游。

{ "envelope": { "message_id": "msg_01J...", "correlation_id": "pt_alg_01J...", "type": "result", // result | error | ack | heartbeat | progress "timestamp": 1780947845 }, "source": { "agent_id": "claude-code-7a...", "task_id": "pt_alg_01J..." }, "payload": { "content_type": "application/json", "data": { "Kp": 2.5, "Ki": 1.0, "Kd": 0.3 } }, "status": "done", "metrics": { "tokens_used": 4523, "elapsed_ms": 45200, "model": "claude-sonnet-4" } }
✅ Schema 门控 Orchestrator 在收到 Envelope JSON 后，使用 PromptTask 中的 output_schema 对 payload.data 进行校验。不通过则按 on_failure 策略处理。
4

Prompt Compiler Agent

Compiler Agent 是一个固定的专用 Claude Code 实例，它的唯一工作是将 WorkflowJSON 编译为一组 PromptTask。它的 Prompt 本身也是固定的——不改变，不升级。

🎯 职责

WorkflowJSON → 拓扑排序 → 模板渲染 → PromptTask[]

🧠 运行时

Claude Code CLI in Docker。claude --print "{compiler_prompt}"

📐 I/O

输入: WorkflowJSON (stdin)
输出: PromptTask[] (stdout)

Compiler Agent 的固定 Prompt（概要）

你是一个 Workflow Compiler Agent。 你的职责是将 WorkflowJSON 编译为一组 PromptTask。 ## 执行步骤 1. 解析 WorkflowJSON — 读取 global_context / nodes[] / edges[] 2. 拓扑排序 — 使用 Kahn 算法，识别 parallel 分支 3. 对每个节点执行模板渲染： a. 查找 templates/{node.type}.yaml b. 按映射注入变量（{global.goal}, {params.xxx}, {output_schema}, {upstream_output}...） c. 生成完整提示词文本 4. 组装 PromptTask — 包含 prompt, output_schema, timeout, retry, depends_on 5. 验证 — prompt 非空、schema 完整、依赖链完整 6. 输出 — 直接输出 PromptTask[] JSON 到 stdout ## 约束 - 严格按照模板格式渲染，不要改写用户原始目标 - 如果模板变量不存在，保留 {变量名} 原文 - 不要添加额外说明或注释
🔄 自洽性质 Compiler Agent 自己也是 Claude Code。AgentFlow 的"Agent 运行在 Docker + Claude Code 中"这一定位，对 Compiler Agent 自身同样成立。它只是被分配了一个不同的 Prompt。
5

模板系统

模板系统是 AgentFlow 的核心资产。每个节点类型对应一个 YAML 模板文件，定义了如何将节点配置渲染为 Agent 提示词。

📂 存储

templates/{node_type}.yaml

🔄 版本管理

每个模板有独立版本号，支持兼容性检查

🔌 可扩展

新增节点类型只需新增一个模板文件

模板示例（algorithm-designer）

# templates/algorithm-designer.yaml name: algorithm-designer version: "1.0.0" description: "算法设计Agent：从需求到控制参数" prompt_template: | ## 目标 {global.goal} → 你需要使用 {params.method} 方法 ## 约束 - 输出必须符合 output_schema: {output_schema} - 全局约束: {global.constraints} - 限时 {timeout_s} 秒 / 重试 {retry} 次 ## 输入上下文 {if upstream_output} 上游 ({upstream_node_label}) 输出: {upstream_output} {endif} ## 输出格式 Envelope JSON, payload 必须符合 output_schema ## 成功标准 payload 通过 output_schema 验证

模板变量系统

变量来源说明
{global.goal}WorkflowJSON.global_context.goal全局目标描述
{global.constraints}global_context.constraints全局约束列表，渲染为 bullet
{params.xxx}nodes[].params.xxx节点特有参数
{output_schema}nodes[].output_schema序列化为 YAML
{upstream_node_label}上游节点 label用于提示词上下文
{upstream_output}上游实际运行时输出编译时注入 schema 样例，运行时注入实际值
{timeout_s}nodes[].timeout_s超时秒数
{retry}nodes[].retry重试次数
6

Orchestrator 调度器

⚡ 职责

接收 PromptTask → 拓扑调度 → docker run → 读 stdout → schema 校验 → 传递

🛡️ 错误处理

超时终止 · 重试逻辑 · 失败回退 · parallel 原子性

📊 状态管理

WAL 持久化 · 心跳检测 · 断点续传 · 画布状态同步

核心循环

步骤动作说明
1拓扑遍历按 sequence 顺序，识别可执行的节点（所有 depends_on 已完成）
2a串行执行单节点：直接创建 Docker 容器
2b并行执行parallel_group 相同：同时创建多个 Docker 容器
3Docker 启动docker run --rm -e KEY={token} claude --print "{prompt}"
4运行时注入将上游实际输出的 payload.data 替换到 {upstream_output} 位置
5监控监听 stdout 解析 Envelope JSON。超时则 docker kill
6验证用 PromptTask.output_schema 校验 payload.data
7a成功保存结果 → 检查是否完成 → 同步画布
7b失败按 retry/on_failure 策略处理
8反馈测试失败 → 重新编译上游 Task → 新容器重试

Docker 容器参数

docker run --rm \ --name agentflow-agent-{task_id} \ --network none \ --cap-drop=ALL \ --security-opt=no-new-privileges:true \ --read-only-root-filesystem \ --tmpfs /tmp:noexec,nosuid,size=100M \ --memory=1G --cpus=2 --pids-limit=200 \ -v agentflow-agent-{task_id}:/workspace \ -e ANTHROPIC_API_KEY="{kms_session_token}" \ agentflow-agent:latest \ claude --print "{prompt}"
🚫 安全铁律 绝不允许 --privileged、--pid=host、--network=host、挂载 Docker Socket。所有密钥通过 KMS 签发限时 Session Token。
7

Agent 运行时

每个 Agent 是一个运行在 Docker 容器中的 Claude Code CLI 实例。通过 stdin 接收完整提示词，通过 stdout 输出 Envelope JSON。

Agent 生命周期

阶段状态说明
① 创建pendingOrchestrator 创建 Docker 容器，注入 KMS session token
② 启动running容器运行中，Claude Code 读取提示词开始处理
③ 完成completedstdout 输出合法 Envelope JSON，exit code=0
④ 失败failed超时 / exit non-zero / schema 校验失败
⑤ 销毁cleaned容器删除，工作区卷清理

Agent 类型注册表

# agent-registry.yaml agent_types: algorithm-designer: description: "算法设计Agent" icon: "🧮" system_prompt: "你是一个专业的控制系统算法设计专家。" default_timeout_s: 120 default_retry: 2 cost_multiplier: 1.0 software-dev: description: "软件开发Agent" icon: "💻" system_prompt: "你是一个专业的嵌入式软件工程师。" default_timeout_s: 300 default_retry: 1 cost_multiplier: 1.5 compile-test: description: "编译测试Agent" icon: "🧪" system_prompt: "你是一个严谨的测试工程师。" default_timeout_s: 180 default_retry: 0 cost_multiplier: 0.8

安全架构

🔑 内部 KMS

Orchestrator 不持有原始 API Key。KMS 签发限时 Session Token，Agent 容器内只消费 Token 不接触 Key。Token 可吊销、可审计。

🐳 容器安全

最小权限原则：cap-drop=ALL · 只读根文件系统 · 独立卷 · 网络隔离 · 资源限制 · 无 Docker Socket 挂载

风险矩阵

风险级别缓解方案
Docker 逃逸🔴 严重最小权限配置，拒绝 privilege／pid=host／socket 挂载
密钥泄露🔴 严重KMS 签发 session token，限时吊销，不外传原始 key
Orchestrator 单点攻破🔴 严重WAL 持久化 + 操作签名 + append-only 审计日志
共享卷跨容器攻击🟡 高每 Agent 独立卷，串联通过 Envelope JSON 不通过文件
API Key 用量炸弹🟡 高token 限时 + 用量跟踪 + 阈值告警
输出泄露敏感信息🟡 高输出内容检查 + 审计日志 + 屏蔽凭据模式

OSS 复用策略

基于「先找，找不到再造」原则，对每个模块的 OSS 评估：

模块分类方案说明
画布 UI🟦 直接复用npm install @xyflow/reactReact Flow 37K⭐，成熟拖拽画布库
Claude SDK🟦 直接复用Anthropic SDK标准 Claude Code CLI，子进程调用
内部 KMS🟩 封装适配HashiCorp Vault + 薄壳层Vault 做密钥后端 + session token 签发
容器编排🟩 封装适配Docker SDK for PythonDocker Engine API，不需要 k8s
Schema 校验🟩 封装适配jsonschema / Ajv标准 JSON Schema 校验库
模板引擎🟩 封装适配Jinja2 / Go template成熟引擎 + AgentFlow 变量绑定层
工作流编译引擎🟥 自研Python/TypeScript核心差异化，无合适 OSS
Orchestrator 调度器🟥 自研PythonAgent 生命周期 + Envelope 协议
模板系统🟥 自研YAML + 渲染器AgentFlow 特有设计
源编排 (L0)🟥 自研Claude Code 驱动LLM 驱动自动生成工作流
📊 总结 10 个核心模块：4 个直接复用 + 3 个封装适配 + 3 个自研。自研部分集中在 AgentFlow 核心差异化价值上。
