Agent Self-Orchestration API

让编排中的 Agent 可以动态调整 DAG 结构、重调度、互相协作，实现自适应闭环

1. 概述

当前 AgentFlow 中，Agent 是被动节点——编排器控制一切。本设计引入 Self-Orchestration API，让 Agent 在运行中可以通过 12 个 REST 端点反控编排系统本身：

当前: 编排器 → Agent（单向控制）
  目标: 编排器 ⟷ Agent（双向对话）

  Agent → 编排器: "Task3 工作量大，我拆成 3 个子任务并行"
  Agent → 编排器: "我的上游产出不够，帮我插入补充分析节点"
  Agent → 编排器: "代码 Review 未通过，请求重置 Task3 重新实现"

设计原则

  • 增量安全 — 默认只允许 Agent 修改自己，需显式授权才能影响其他节点
  • 无事务冲突 — 每个 Agent 的 API 调用在前一个节点完成后生效，不会出现两个 Agent 同时改同一节点
  • 轻量注入 — API 信息通过环境变量 + 文件 + system prompt 三条路径注入，不改 Agent 运行时内核
  • 可观测 — 所有 API 调用记录到 run_events，可在前端实时看到 DAG 变更

2. API 设计

共 12 个端点，分为 3 类。所有端点路径以 /api/runs/{run_id} 为前缀。

2.1 只读 — 获取状态

方法路径用途响应
GET /api/runs/{rid} 完整运行状态 status, nodes[], total_cost 等
GET /api/runs/{rid}/nodes 获取节点拓扑 所有节点含上下游 ID
GET /api/runs/{rid}/nodes/{nid} 单节点详情 result, output, cost, turns, params
GET /api/runs/{rid}/edges DAG 边列表 [{source, target}, ...]

2.2 写操作 — 动态调整 DAG

方法路径用途请求体
POST /api/runs/{rid}/nodes 新增节点到 DAG {id, label, profile, desc, params, depends_on[]}
PATCH /api/runs/{rid}/nodes/{nid} 修改节点 {desc?, params?, profile?, ...}
DELETE /api/runs/{rid}/nodes/{nid} 删除节点及下游 —
POST /api/runs/{rid}/edges 新增依赖边 {source, target}
DELETE /api/runs/{rid}/edges 删除依赖边 {source, target}
POST /api/runs/{rid}/nodes/{nid}/retry 重置节点重跑 {modify_desc?: str}
POST /api/runs/{rid}/reroute 批量调整 DAG {add_nodes[], remove_nodes[], add_edges[], remove_edges[]}

2.3 控制流

方法路径用途请求体
POST /api/runs/{rid}/feedback Agent 提反馈给编排器 {from_node, type, message, suggested_action?, target_nodes?}

统一响应格式

{
  "ok": true,
  "data": { ... },       // 仅成功时
  "event_id": "ev_xxx",  // 写入 run_events 的事件 ID
  "dag_version": 3       // DAG 版本号（用于下游检测变更）
}

// 错误
{
  "ok": false,
  "error": "scope_denied: 无权修改上游节点 task1",
  "code": "SCOPE_DENIED"
}

3.3 System Prompt 注入

在 PromptCompiler 编译任务时，自动附加 Self-Orchestration 能力描述：

## 编排 API 能力
你可以通过以下 API 动态调整工作流。API 的基础 URL 和鉴权信息在环境变量和工作目录中。

### 可用操作
1. **查看状态** — GET /api/runs/ 查看整体进度
2. **查看同行** — GET /api/runs//nodes 看所有节点
3. **新增节点** — POST /api/runs//nodes 添加子任务
4. **修改自己** — PATCH /api/runs//nodes/<自己的节点ID>

4. 运行时 DAG 突变机制

当前 _execute_run 的 while 循环是"读一次就运行"模式。改造后引入 DAG 版本号 + 增量检测：

_execute_run(run_id):
  work_dir = _run_workspace(run_id)
  dag_version = 0

  while not all_nodes_done:
    # 检查 DAG 是否有外部变更
    current_dag_version = get_dag_version(run_id)    ← 新
    if current_dag_version > dag_version:
      apply_dag_mutations(run_id, dag_version)        ← 新
      dag_version = current_dag_version

    ready = get_pending_nodes()
    if not ready:
      # deadlock 检测（不变）
      break

    for rn in ready:
      # 执行节点的过程中，Agent 随时可能通过 API 修改 DAG
      result = execute_with_agentflow_api(rn, ...)
      # 每执行完一个节点，检查版本号
      if get_dag_version(run_id) > dag_version:
        break  # 跳出循环，重新评估 ready 集合

apply_dag_mutations 做什么

  1. 从 DB 读取最新 DAG 快照（workflow_snapshots 表的 version 字段）
  2. 与内存中的 node_map / edge_map 对比差异
  3. 新增节点 — 插入 node_map，状态 pending，在下次循环中被 get_pending_nodes 拾取
  4. 修改节点 — 更新 node_map 中的 desc/params/profile
  5. 删除节点 — 标记 skipped，从 node_map 移除，释放依赖
  6. 新增边 — 插入 edge_map，触发依赖重新评估
  7. 删除边 — 从 edge_map 移除

DAG 版本号实现

在 runs 表新增 dag_version INTEGER DEFAULT 0 字段。每次写操作（add/delete/patch node/edge）自动递增。查询时原子读取：

-- 在事务中
UPDATE runs SET dag_version = dag_version + 1 WHERE run_id = ?
SELECT dag_version FROM runs WHERE run_id = ?

一致性保证

  • Agent 的 API 调用和后台 worker 的 while 循环共享同一个 SQLite DB（单线程串行执行，天然无竞争）
  • Agent 添加的节点若依赖已完成的节点 → 立即加入 ready 集合（自动执行）
  • Agent 删除自己时，编排器自动重新连接其上下游（父→子，跳过删除节点）
  • 删除节点时，其下游所有节点变为 ready（不再依赖已删除的节点）

5. Scope 安全模型

scope改自己改下游改上游改并行适用场景
self（默认）✅❌❌❌普通 dev/test 节点
downstream✅✅❌❌架构师/设计节点
run✅✅✅✅Supervisor/编排节点

Token 生成

后端启动时生成临时 token，编码了 scope 信息：

token = base64(f"{run_id}:{node_id}:{scope}:{hmac(secret)}")

所有写操作端点验证 token：

def check_scope(token, target_node_id, operation):
  payload = decode_token(token)
  if payload.scope == "run":
    return True
  if payload.scope == "self":
    return target_node_id == payload.node_id
  if payload.scope == "downstream":
    return is_downstream(payload.node_id, target_node_id, dag)
  return False

Feedback 特殊处理

POST /api/runs/{rid}/feedback 不检查 scope——任何节点都可以给编排器提建议。编排器根据建议内容决定是否执行操作。

6. 使用场景

场景 1：Agent 发现上游不完整，动态补充

task3（开发）执行到一半发现 task2（设计）只给了文字描述没有代码骨架。

task3 调用: POST /api/runs/xxx/nodes
  {
    "id": "task2.5",
    "label": "补充设计",
    "profile": "design",
    "desc": "生成代码骨架和接口定义",
    "depends_on": ["task2"]
  }
task3 调用: POST /api/runs/xxx/edges
  {"source": "task2.5", "target": "task3"}
→ 编排器：新节点 task2.5 已插入，
  task3 变为依赖 task2.5 和 task2，
  task2.5 自动开始执行

场景 2：Agent 自我并行化

task3分析发现要写 3 个独立模块（串口/UI/导出）。

task3 调用: POST /api/runs/xxx/reroute
  {
    "remove_nodes": ["task3"],
    "add_nodes": [
      {"id": "task3a", "label": "串口模块", "desc": "...", "depends_on": ["task2"]},
      {"id": "task3b", "label": "UI模块", "desc": "...", "depends_on": ["task2"]},
      {"id": "task3c", "label": "导出模块", "desc": "...", "depends_on": ["task2"]},
      {"id": "task3_merge", "label": "集成汇总", "desc": "...",
       "depends_on": ["task3a", "task3b", "task3c"]}
    ],
    "add_edges": [
      {"source": "task3_merge", "target": "task4"}
    ]
  }
→ 3 个模块并行执行，全部完成后 merge 节点汇总，再触发 task4

场景 3：Agent 间 Review + Retry

task4 Review 了 task3 的代码，发现质量不达标。

task4 调用: POST /api/runs/xxx/feedback
  {
    "from_node": "task4",
    "type": "review_failed",
    "message": "task3 的串口模块超时处理缺失，建议补充",
    "suggested_action": "retry",
    "target_nodes": ["task3"]
  }

编排器收到 feedback:
  1. 检查 task3 的原 prompt
  2. 追加 task4 的 review 意见作为 retry_prompt
  3. 调用 POST /api/runs/xxx/nodes/task3/retry
     {"modify_desc": "补充超时处理：...原 prompt..."}
  4. task4 保持 pending 直到 task3 重新完成
