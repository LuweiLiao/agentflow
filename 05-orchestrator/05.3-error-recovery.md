# 05.3 — 错误恢复机制

## 1. 概述

AgentFlow 的错误恢复机制确保长时间运行的工作流在面对网络抖动、LLM 服务故障、资源耗尽等异常情况时，能够自动恢复而不丢失进度。设计核心围绕 **三阶段防御** 展开：

| 阶段 | 机制 | 目标 |
|------|------|------|
| **预防** | 超时检测 + 资源限制 | 在错误发生前限制影响范围 |
| **恢复** | 重试 + 重编译 + 回滚 | 自动修复可恢复的错误 |
| **持久化** | WAL 日志 + 断点续传 | 即使进程崩溃也不丢进度 |

## 2. 错误分类

| 错误类型 | 错误来源 | 可恢复性 | 默认策略 |
|----------|---------|---------|---------|
| `TimeoutError` | 容器超时 | ✅ 可重试 | 重试 3 次，间隔 10s |
| `OOMKilled` | Docker OOM Killer | ✅ 可重试 | 增加内存后重试 2 次 |
| `APIError` | LLM 服务返回 5xx | ✅ 可重试 | 指数退避重试 5 次 |
| `AuthError` | API Key 无效 / 过期 | ❌ 不可恢复 | 立即中止，通知用户 |
| `ValidationError` | 输出不匹配 schema | ⚠️ 可反馈 | 触发反馈循环重编译 |
| `CycleDependencyError` | 依赖图有环 | ❌ 不可恢复 | 编译阶段即报错 |
| `ContainerError` | 镜像拉取失败 / 启动失败 | ⚠️ 有限重试 | 重试 2 次，切换镜像 |
| `DiskFullError` | 磁盘空间不足 | ❌ 不可恢复 | 中止工作流，告警 |
| `UnknownError` | 未分类异常 | ⚠️ 尝试重试 | 重试 1 次后中止 |

## 3. 超时机制

### 配置

每个 PromptTask 可以设置独立的超时时间：

```yaml
task:
  id: "task-003"
  timeout_seconds: 300   # 5 分钟超时
  retry:
    on_timeout: "retry"
    count: 3
    delay_seconds: 10
```

### 实现

```python
def monitor_with_timeout(container, timeout_seconds):
    """
    监控容器 stdout，并在超时时强制 kill。
    
    超时检测策略：
    - 从容器启动开始计时
    - 每次收到 stdout 行时刷新"最后活跃时间"
    - 当前时间 - 最后活跃时间 > timeout_seconds 则视为超时
    """
    last_active = time.monotonic()
    start_time = time.monotonic()
    
    for line in container.logs(stream=True, decode=True):
        if line.strip():
            last_active = time.monotonic()
        
        elapsed = time.monotonic() - start_time
        if elapsed > timeout_seconds:
            container.kill()
            raise TimeoutError(
                f"Task 执行超时 ({timeout_seconds}s)，"
                f"最后活跃于 {elapsed - (time.monotonic() - last_active):.1f}s 前"
            )
```

### 超时阶梯

| 超时类型 | 触发条件 | 行为 |
|---------|---------|------|
| **静默超时** | 容器启动后无任何输出 > 60s | 重试，可能是容器卡死 |
| **执行超时** | 总运行时间 > `timeout_seconds` | 按 `on_timeout` 策略处理 |
| **空闲超时** | 最后输出后静默 > `idle_timeout` (默认 120s) | 认为 LLM 卡住，kill 重试 |

## 4. 重试策略

### 重试配置

```yaml
task:
  retry:
    count: 3                     # 最大重试次数
    delay_seconds: 10            # 固定等待秒数
    backoff_multiplier: 2.0      # 退避倍数（可选）
    max_delay_seconds: 120       # 最大退避间隔
    jitter: true                 # 是否启用抖动
    retryable_errors:            # 仅重试指定错误
      - TimeoutError
      - APIError
      - ContainerError
```

### 退避算法

```python
def calculate_delay(attempt, base_delay, multiplier, max_delay, jitter):
    """
    指数退避 + 抖动。
    
    公式: delay = min(base_delay * multiplier^attempt, max_delay)
    抖动: delay = delay * (0.5 + random())   # 0.5x ~ 1.5x 随机化
    """
    delay = min(base_delay * (multiplier ** attempt), max_delay)
    if jitter:
        delay *= 0.5 + random.random()
    return delay
```

示例退避时间线（base=2s, multiplier=2, max_delay=120s, jitter=true）：

| 重试次数 | 理论延迟 | 实际延迟（含抖动） |
|---------|---------|-----------------|
| 1 | 2s | 1.0s ~ 3.0s |
| 2 | 4s | 2.0s ~ 6.0s |
| 3 | 8s | 4.0s ~ 12.0s |
| 4 | 16s | 8.0s ~ 24.0s |
| 5 | 32s | 16.0s ~ 48.0s |
| 6 | 64s | 32.0s ~ 96.0s |
| 7+ | 120s | 60.0s ~ 180.0s |

## 5. 回滚机制

### 回滚触发条件

- 重试耗尽后仍失败，且 `on_failure: "rollback"`
- 下游 Task 因上游数据不一致而无法执行
- 用户手动触发回滚

### 回滚流程

```
1. 定位回滚点
   └─ 找到最近的成功检查点（checkpoint）
   
2. 恢复上下文
   └─ 从 WAL 读取检查点的完整状态
   
3. 重建容器
   └─ 从检查点重新启动所有 Task
   
4. 增量执行
   └─ 仅执行回滚点之后未完成的 Task
```

### 增量回滚

```python
def rollback_to_checkpoint(checkpoint_id, current_state):
    """
    回滚到指定检查点，支持增量恢复。
    
    注意：不重新执行已成功的 Task，只重做回滚点之后失败的 Task。
    """
    checkpoint = wal.load_checkpoint(checkpoint_id)
    
    # 1. 恢复上下文到检查点状态
    rollback_context = checkpoint.context
    
    # 2. 找出检查点之后新增/变更的 Task
    rollback_tasks = []
    for task in current_state.pending_tasks:
        if task.id not in checkpoint.completed_tasks:
            rollback_tasks.append(task)
    
    # 3. 清除检查点之后的副作用（如已同步的画布数据）
    rollback_side_effects(checkpoint_id)
    
    # 4. 重新执行
    return orchestrator_core_loop(rollback_tasks, rollback_context)
```

## 6. WAL 预写日志

### 设计原理

WAL（Write-Ahead Log）是所有状态变更的先写日志，确保 AgentFlow 进程在任何时刻崩溃，重启后都能从最后一个完整记录中恢复。

```
普通写入：
  内存修改 → 写入数据库
  （崩溃 → 数据丢失）

WAL 写入：
  追加日志 → 内存修改 → 写入数据库
  （崩溃 → 重放日志恢复）
```

### WAL 日志格式

每条 WAL 记录是一个 JSON 行，追加写入日志文件：

```json
{"wal_seq": 1, "ts": "2026-06-09T10:21:00Z", "type": "task_start",   "task_id": "t1", "session_id": "s-001"}
{"wal_seq": 2, "ts": "2026-06-09T10:21:05Z", "type": "task_chunk",   "task_id": "t1", "seq": 1, "data_len": 256}
{"wal_seq": 3, "ts": "2026-06-09T10:21:08Z", "type": "task_final",   "task_id": "t1", "status": "success", "checksum": "a1b2c3"}
{"wal_seq": 4, "ts": "2026-06-09T10:21:10Z", "type": "task_start",   "task_id": "t2", "session_id": "s-001"}
{"wal_seq": 5, "ts": "2026-06-09T10:21:20Z", "type": "task_error",   "task_id": "t2", "error": "TimeoutError", "attempt": 1}
{"wal_seq": 6, "ts": "2026-06-09T10:21:35Z", "type": "task_retry",   "task_id": "t2", "attempt": 2}
{"wal_seq": 7, "ts": "2026-06-09T10:21:40Z", "type": "checkpoint",   "checkpoint_id": "cp-001", "completed": ["t1"]}
```

| 事件类型 | 说明 |
|---------|------|
| `task_start` | Task 开始执行 |
| `task_chunk` | 收到部分输出（仅记录长度和 checksum，不存全量数据） |
| `task_final` | Task 成功完成 |
| `task_error` | Task 执行出错 |
| `task_retry` | Task 开始重试 |
| `task_skip` | Task 被跳过 |
| `checkpoint` | 检查点记录 |
| `rollback` | 回滚事件 |
| `workflow_done` | 工作流完成 |
| `workflow_abort` | 工作流被中止 |

### WAL 存储结构

```
/var/agentflow/wal/
├── sessions/
│   ├── s-001.wal           # 每个会话一个 WAL 文件
│   └── s-002.wal
├── checkpoints/
│   ├── cp-001.json         # 检查点快照
│   └── cp-002.json
└── wal_index.json          # WAL 文件索引
```

### WAL 重放

```python
def recover_from_wal(session_id):
    """
    从 WAL 恢复会话状态。
    
    流程：
    1. 找到会话的最后一个 WAL 文件
    2. 从末尾向前扫描，找到最近的 checkpoint
    3. 从 checkpoint 之后的记录开始重放
    4. 重建未完成 Task 的上下文
    """
    wal_path = f"/var/agentflow/wal/sessions/{session_id}.wal"
    
    # 1. 扫描所有记录
    records = []
    with open(wal_path, 'r') as f:
        for line in f:
            records.append(json.loads(line))
    
    # 2. 找到最后一个 checkpoint
    last_checkpoint = None
    for record in reversed(records):
        if record["type"] == "checkpoint":
            last_checkpoint = record
            break
    
    # 3. 从 checkpoint 之后重放
    replay_start = 0
    if last_checkpoint:
        checkpoint_state = load_checkpoint(last_checkpoint["checkpoint_id"])
        replay_start = last_checkpoint["wal_seq"]
    
    # 4. 重建未完成状态
    pending_tasks = []
    for record in records[replay_start:]:
        if record["type"] == "task_start":
            pending_tasks.append(record["task_id"])
        elif record["type"] == "task_final" or record["type"] == "task_error":
            pending_tasks = [t for t in pending_tasks if t != record["task_id"]]
    
    return pending_tasks
```

## 7. 断点续传

### 检查点创建

Orchestrator 在以下时机自动创建检查点：

- **每完成一个并行批次**（一个 `parallel_group` 所有节点完成）
- **每完成 N 个 Task**（`save_checkpoint_every: 10`）
- **用户手动触发**（API 调用）

### 检查点快照内容

```json
{
  "checkpoint_id": "cp-042",
  "session_id": "s-001",
  "workflow_id": "wf-007",
  "created_at": "2026-06-09T10:21:00Z",
  "completed_tasks": ["t1", "t2", "t3"],
  "failed_tasks": ["t4"],
  "skipped_tasks": [],
  "context": {
    "upstream_outputs": {
      "t1": {"checksum": "abc", "summary": "..."},
      "t2": {"checksum": "def", "summary": "..."}
    },
    "env_snapshot": {
      "API_KEY_ALIAS": "sk-...",
      "WORKSPACE": "/workspace/wf-007"
    }
  }
}
```

> 注意：检查点不保存完整输出数据，只保存 checksum 和摘要。完整数据从 WAL 的 `task_final` 记录重建。

### 断点续传流程

```
用户重新提交工作流
        │
        ▼
检测到 session_id 已存在
        │
        ▼
┌─────────────────────────┐
│  WAL 恢复                │
│  → 找到最后一个 checkpoint │
│  → 重放增量记录           │
│  → 重建未完成 Task 列表   │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  跳过已完成 Task         │
│  → 直接从 checkpoint     │
│  → 恢复上游输出上下文     │
└──────────┬──────────────┘
           │
           ▼
┌─────────────────────────┐
│  继续执行剩余 Task       │
│  → 从断点处进入核心循环   │
└─────────────────────────┘
```

## 8. 通知与告警

### 通知事件

| 事件 | 通知方式 | 说明 |
|------|---------|------|
| 工作流开始 | 画布状态更新 | 开始执行的第一个 Task |
| Task 重试 | 日志 + 可选 Webhook | 每次重试发送 |
| Task 失败（耗尽重试） | 日志 + Webhook | 通知运维人员 |
| 工作流完成 | 画布状态更新 + Webhook | 携带结果摘要 |
| 工作流中止 | 紧急通知（即时） | 不可恢复错误 |

### Webhook 回调格式

```json
{
  "event": "task_retry_exhausted",
  "workflow_id": "wf-007",
  "session_id": "s-001",
  "task_id": "t4",
  "error": "TimeoutError",
  "attempts": 3,
  "timestamp": "2026-06-09T10:21:35Z",
  "recovery_action": "rollback"
}
```

## 9. 配置参考

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `retry.count` | 3 | 最大重试次数 |
| `retry.delay_seconds` | 10 | 重试基础等待时间 |
| `retry.backoff_multiplier` | 2.0 | 指数退避倍数 |
| `retry.max_delay_seconds` | 120 | 最长退避间隔 |
| `retry.jitter` | `true` | 启用随机抖动 |
| `timeout_seconds` | 300 | Task 执行超时 |
| `idle_timeout_seconds` | 120 | 空闲超时 |
| `save_checkpoint_every` | 10 | 每 N 个 Task 创建检查点 |
| `wal.path` | `/var/agentflow/wal/` | WAL 存储路径 |
| `wal.flush_interval` | `1s` | WAL flush 到磁盘的间隔 |
| `webhook.url` | — | 告警 Webhook 地址 |
| `max_feedback_depth` | 3 | 反馈循环最大深度 |

## 10. 典型恢复场景

### 场景 1：LLM API 临时不可用

```
1. Task t3 调用 LLM API 返回 503
2. Orchestrator 捕获 APIError
3. 检查 retryable_errors，确认 APIError 可重试
4. 等待 2s（指数退避第 1 次），重试
5. 再次 503，等待 4s，重试
6. 第 3 次成功，继续执行
```

### 场景 2：容器 OOM

```
1. Task t5 内存超限，Docker OOM Killer 终止容器
2. Orchestrator 检测 OOMKilled 退出码
3. 读取 Docker 事件确认 OOM
4. 自动增加内存限制（4g → 6g）
5. 重新启动容器重试
6. 如果仍然 OOM，继续增加内存直到上限
```

### 场景 3：AgentFlow 进程崩溃后恢复

```
1. AgentFlow 进程意外退出（SIGKILL / 机器重启）
2. 重新启动 AgentFlow 服务
3. 启动时扫描 /var/agentflow/wal/ 目录
4. 发现 s-001 会话有未完成的 WAL 记录
5. 从最后一个 checkpoint cp-003 恢复
6. 重放 checkpoint 之后的 5 条增量记录
7. 重建上下文，确定 t8 正在执行、t9 尚未开始
8. 从 t8 继续执行，无需重新运行 t1-t7
```

### 场景 4：下游测试失败触发反馈重试

```
1. Task t10（验证 Agent）对 t9 的输出测试失败
2. Orchestrator 收到 test_result: {status: "fail", feedback: "..."}
3. 进入反馈循环（步骤 ⑧）
4. 重新编译 t9 的 prompt template，加入反馈
5. 为 t9 启动新容器
6. t9 新输出通过 t10 测试，继续执行
7. 如果 3 次反馈仍失败，标记为不可恢复错误
```
