# 05.2 — Docker 容器生命周期管理

## 1. 概述

AgentFlow 的每个 PromptTask 在一个独立 Docker 容器中执行，容器是运行时隔离的核心单元。容器生命周期由 Orchestrator 全程管理，包括镜像选择、容器创建、参数注入、运行监控和销毁回收。

```
         ┌──────────────┐
         │  镜像构建     │
         │  (构建时)     │
         └──────┬───────┘
                │
                ▼
         ┌──────────────┐
         │  容器创建     │
         │  docker run   │
         └──────┬───────┘
                │
                ▼
         ┌──────────────┐
         │  运行时注入   │
         │  环境变量/卷  │
         └──────┬───────┘
                │
         ┌──────┴──────┐
         │             │
         ▼             ▼
   ┌──────────┐  ┌──────────┐
   │ 正常运行  │  │ 超时/异常 │
   │ stdout流  │  │ docker kill
   └────┬─────┘  └────┬─────┘
        │             │
        └──────┬──────┘
               │
               ▼
         ┌──────────────┐
         │  容器销毁     │
         │  --rm / rm   │
         └──────────────┘
```

## 2. 镜像构建

### 基础镜像

AgentFlow 使用预构建的 Docker 镜像，每个镜像封装特定的语言模型运行环境。

| 镜像标签 | 基础环境 | 默认模型 | 用途 |
|----------|---------|---------|------|
| `claude:latest` | Python 3.12 + Claude CLI | Claude Opus | 通用文本推理 |
| `claude:code` | Python 3.12 + Claude Code CLI | Claude Sonnet | 代码生成与执行 |
| `gpt:latest` | Python 3.12 + OpenAI SDK | GPT-4o | OpenAI 兼容推理 |
| `agent:custom` | 用户自定义 Dockerfile | — | 扩展运行时 |

### 构建流程

```
# 自动构建 Dockerfile
FROM python:3.12-slim

# 安装运行时依赖
RUN pip install --no-cache-dir \
    claude-cli>=2.0.0 \
    jsonschema>=4.0

# 设置工作目录
WORKDIR /workspace

# 默认入口：从环境变量读取 prompt 并执行
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

构建命令：

```bash
docker build -t claude:latest -f Dockerfile.claude .
```

### 镜像缓存策略

- 基础镜像层缓存于本地 Docker 守护进程
- 仅当 `requirements.txt` 或 `entrypoint.sh` 变更时重新构建
- 生产环境建议推送到私有镜像仓库（`registry.agentflow.io/`）

## 3. 容器运行参数

### 完整 `docker run` 命令

```bash
docker run \
  --rm \
  --name "agentflow-{task_id}" \
  -e HERMES_API_KEY="sk-..." \
  -e TASK_ID="{task.id}" \
  -e SEQUENCE="{task.sequence}" \
  -e PARALLEL_GROUP="{task.parallel_group}" \
  -e AGENTFLOW_SESSION_ID="{session_id}" \
  -e AGENTFLOW_WORKFLOW_ID="{workflow_id}" \
  -e AGENTFLOW_UPSTREAM_DATA="{上游输出(JSON编码)}" \
  -v "{workspace_dir}:/workspace:ro" \
  -v "{cache_dir}:/cache:rw" \
  -v "{wal_dir}:/var/agentflow:rw" \
  -w /workspace \
  --network "agentflow-net" \
  --memory "4g" \
  --cpus "2" \
  --memory-swap "6g" \
  --ulimit nofile=1024:2048 \
  --stop-timeout 30 \
  --log-driver "json-file" \
  --log-opt max-size="10m" \
  --log-opt max-file="3" \
  claude:latest \
  claude --print "{prompt_template}"
```

### 参数分类说明

#### 3.1 运行时标识

| 参数 | 说明 |
|------|------|
| `--name` | 容器名 = `agentflow-{task.id}`，便于运维定位 |
| `--rm` | 退出后自动删除容器，避免残留 |

#### 3.2 环境变量注入

| 环境变量 | 来源 | 说明 |
|----------|------|------|
| `HERMES_API_KEY` | 系统配置 / 用户 Token | LLM API 鉴权 |
| `TASK_ID` | PromptTask.id | 当前 Task 标识 |
| `SEQUENCE` | PromptTask.sequence | 拓扑层级 |
| `PARALLEL_GROUP` | PromptTask.parallel_group | 并行分组 |
| `AGENTFLOW_SESSION_ID` | 会话 ID | 用于日志关联 |
| `AGENTFLOW_WORKFLOW_ID` | 工作流 ID | 用于断点续传 |
| `AGENTFLOW_UPSTREAM_DATA` | 上游 Task 结果 | JSON 编码的上游 payload.data |

**敏感信息处理：**

- `HERMES_API_KEY` 等敏感变量从 AgentFlow 安全存储（Vault / 加密文件）读取
- 运行时通过 `docker run -e` 直接注入，不写入磁盘
- 日志中自动遮盖敏感变量值（`***`）

#### 3.3 卷挂载

| 挂载路径 | 宿主机路径 | 模式 | 用途 |
|----------|-----------|------|------|
| `/workspace` | `{workspace_dir}` | `ro` | 只读工作区，包含任务输入文件和 prompt |
| `/cache` | `{cache_dir}` | `rw` | 模型缓存、临时文件 |
| `/var/agentflow` | `{wal_dir}` | `rw` | WAL 日志输出、检查点写入 |

**卷管理最佳实践：**

- **工作区只读**：防止容器内任务意外篡改输入文件
- **缓存可写**：LLM 模型缓存需要写入权限
- **WAL 日志持久化**：即使容器被 kill，WAL 日志仍保留在宿主机

#### 3.4 资源限制

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `--memory` | `4g` | 内存硬限制 |
| `--memory-swap` | `6g` | 内存 + Swap 上限 |
| `--cpus` | `2` | CPU 核心数 |
| `--ulimit nofile` | `1024:2048` | 文件描述符限制 |
| `--stop-timeout` | `30` | `docker stop` 等待时间（秒） |

> 资源限制值可通过 PromptTask 的 `resources` 字段覆盖。

#### 3.5 网络配置

```bash
--network "agentflow-net"
```

- 所有 Task 容器加入同一个 bridge 网络 `agentflow-net`
- 容器间通过容器名互联（`agentflow-{task.id}`）
- 可选的 `--publish` 端口映射暴露 HTTP 服务

#### 3.6 日志驱动

```bash
--log-driver "json-file"
--log-opt max-size="10m"
--log-opt max-file="3"
```

- 日志最多保留 3 个文件，每个 10MB
- 超出自动轮转

## 4. 容器内标准输出协议

### Envelope JSON 行协议

容器内运行的程序必须按行输出标准格式的 JSON，每行一个 Envelope：

```
{"type":"chunk","seq":1,"payload":{"data":"..."}}
{"type":"final","seq":2,"payload":{"data":"...","metadata":{}}}
```

更多细节见 [05.1-core-loop.md 步骤 ⑤](./05.1-core-loop.md#步骤-⑤监控-stdout解析-envelope-json)。

### 非 JSON 输出处理

- 非 JSON 行作为原始文本记录，不参与结构化解析
- 仅 `type: "final"` 的 Envelope 参与结果校验和保存
- `type: "error"` 的 Envelope 立即触发错误处理

## 5. 容器生命周期钩子

Orchestrator 在容器的关键生命周期点触发钩子事件：

| 阶段 | 钩子事件 | 说明 |
|------|---------|------|
| 创建前 | `before_container_start` | 校验镜像是否存在，拉取最新版本 |
| 启动后 | `after_container_start` | 记录容器 ID，注册超时定时器 |
| stdout 首行 | `on_first_output` | 标记容器已开始工作 |
| 正常退出 | `on_container_exit` | 收集退出码，清理资源 |
| 超时 kill | `on_timeout_kill` | 记录超时日志，触发错误恢复 |
| 异常 OOM | `on_oom_kill` | 检查 OOM Killer 日志，增加资源重试 |

## 6. 容器安全

### 安全策略

- **非 root 运行**：容器内以 `agentflow` 用户（UID 1000）运行
- **只读根文件系统**：`--read-only` 强制根文件系统只读
- **禁止 privilege**：`--security-opt no-new-privileges:true`
- **能力裁剪**：`--cap-drop ALL --cap-add NET_BIND_SERVICE`
- **Seccomp 限制**：使用默认 seccomp 策略，禁用不必要的系统调用

### 安全配置示例

```bash
docker run \
  --rm \
  --read-only \
  --user 1000:1000 \
  --security-opt no-new-privileges:true \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --security-opt seccomp=default.json \
  ...
```

## 7. 清理与回收

### 正常清理

- `--rm` 标志确保退出时自动删除容器
- Orchestrator 在得到 `type: "final"` 或 `type: "error"` 后等待最多 5 秒让容器自然退出

### 异常清理

```python
def force_cleanup(container_id):
    try:
        container = docker_client.containers.get(container_id)
        container.kill()       # SIGKILL
        container.remove(force=True)  # 强制删除
    except docker.errors.NotFound:
        pass  # 容器已不存在，无需处理
    
    # 清理残留卷
    docker_client.volumes.prune(filters={"label": f"task={task_id}"})
```

### 孤儿容器兜底

每隔 60 秒执行一次兜底扫描：

```bash
docker ps -a --filter "name=agentflow-*" --filter "status=exited" -q | xargs -r docker rm
docker ps -a --filter "name=agentflow-*" --filter "status=created" -q | xargs -r docker rm
```
