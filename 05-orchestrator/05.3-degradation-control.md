# 05.3 — 降级机制与超时控制

## 1. 概述

Orchestrator 的降级机制确保工作流在资源受限、部分节点失败或异常情况下仍然能够产出有意义的输出。三阶段防御模型：

| 阶段 | 机制 | 目标 |
|------|------|------|
| **预防** | 模板超时 + max_turns 限制 | 在错误发生前限制影响范围 |
| **降级** | 模板回退 + 错误结果封装 + 节点隔离 | 失败节点不影响全局 |
| **兜底** | 工作目录清理 + 结果截断 + 响应聚合 | 保证 API 总是返回有效 JSON |

```
正常路径:
  Nodes[0] ──→ Nodes[1] ──→ Nodes[2] ──→ 返回结果
     │             │             │
     ▼             ▼             ▼
    OK            OK            OK

降级路径（节点2失败）:
  Nodes[0] ──→ Nodes[1] ──→ Nodes[2](error) ──→ 返回结果
     │             │             │                  │
     ▼             ▼             ▼                  ▼
    OK            OK        {status:error}    Nodes[2] 标记为失败
                                              Nodes[0,1] 结果保留
                                              不中断工作流
```

## 2. 降级机制

### 2.1 模板加载降级

当节点的 `profile` 对应的 YAML 模板不存在时，自动回退到默认模板：

```python
def load(self, profile: str) -> dict:
    path = os.path.join(self.template_dir, f"{profile}.yaml")
    if not os.path.isfile(path):
        # 模板不存在 → 回退到 dev.yaml
        fallback = os.path.join(self.template_dir, "dev.yaml")
        if os.path.isfile(fallback):
            path = fallback
        else:
            raise TemplateNotFoundError(f"模板不存在: {path}")
    ...
```

**效果：** 即使 `deploy.yaml` 被误删，使用 `deploy` profile 的节点也会用 `dev.yaml` 渲染——虽然角色不精确，但工作流不会中断。

### 2.2 变量渲染降级

当模板变量不存在时，保留原始占位符：

```python
def replace_var(match):
    key = match.group(1)
    # ...
    # 变量不存在 → 返回原始 {key}
    return f"{{{key}}}"
```

**效果：** 如果 `{params.custom_field}` 在变量映射中不存在，渲染结果保留 `{params.custom_field}` 原文。Agent 看到的是一个占位符，通常会自动忽略或解释。

### 2.3 节点级错误隔离

```python
for fut in as_completed(fut_to_node):
    node = fut_to_node[fut]
    try:
        result = fut.result()
    except Exception as e:
        # 单个节点异常 → 封装为错误结果
        result = {
            "status": "error",
            "result": f"执行异常: {e}",
            "cost": 0, "duration": 0, "turns": 0,
        }
    # 无论成功还是失败，都更新节点
    node.status = result.get("status", "error")
    node.result = result.get("result", "")
    all_results.append(node.to_dict())
```

**关键特性：**
- `fut.result()` 抛异常 → 只影响当前 `Future`
- 同组其他 `Future` 继续执行，不受影响
- 异常节点标记为 `error`，但结果仍然包含在响应中
- 前端可以展示所有节点结果，即使部分失败

### 2.4 LLM 模型降级

```python
# handle_decompose 中的 LLM 调用降级
try:
    nodes = self.call_llm(requirement, count)
except Exception as e:
    # LLM 调用失败 → 使用本地模板生成
    nodes = self.fallback_template(requirement, count)
```

**效果：** 如果智谱 API 不可用，`/api/decompose` 自动使用内置的 `fallback_template()` 生成节点列表，保证前端画布始终可用。

### 2.5 输出截断降级

```python
node.output = result.get("output", "")[:2000]  # 截断前 2000 字符
```

**效果：** 避免过大输出撑爆 HTTP 响应体。前端显示时若发现输出被截断，可提示"输出过长，已截断"。

## 3. 超时控制层次

### 3.1 三层超时模型

| 层级 | 限制参数 | 默认值 | 说明 |
|------|---------|--------|------|
| **LLM 调用层** | API 请求超时 | 60s（由 AgentRunner 内部设置） | 单次 LLM 调用超时 |
| **Agent 执行层** | `timeout_s` | 120–300s（由模板定义） | 整个 Agent 循环超时 |
| **HTTP 服务层** | HTTP 请求超时 | 依赖客户端设置 | 前端等待响应超时 |

### 3.2 Agent 执行层超时

`AgentRunner.execute()` 内部实现超时检查：

```python
class AgentRunner:
    def execute(self, prompt, work_dir, profile,
                tools_enabled=True, max_turns=10, timeout=120):
        start_time = time.time()
        turns = 0

        while turns < max_turns:
            elapsed = time.time() - start_time
            if elapsed > timeout:
                return {
                    "status": "timeout",
                    "result": f"执行超时 ({timeout}s)",
                    "output": "\n".join(output_lines),
                    "cost": total_cost,
                    "duration_ms": int(elapsed * 1000),
                    "turns": turns,
                }

            # Think → Act (call tool) → Observe
            ...
            turns += 1

        return {"status": "ok", ...}
```

### 3.3 max_turns 限制

`max_turns` 防止 Agent 陷入无限循环：

| 模板 | max_turns | 典型场景 |
|------|-----------|---------|
| 10 | 分析/设计/文档 | 一次性输出，不需要多轮迭代 |
| 15 | 开发/测试/部署 | 需要编码→调试→测试多轮迭代 |

**超出 max_turns 的行为：**
```python
if turns >= max_turns:
    return {
        "status": "ok",  # 仍然算成功
        "result": f"达到最大轮数 {max_turns}，自动结束",
        ...
    }
```

达到 `max_turns` 后 Agent 自动停止，当前累积的输出作为最终结果返回。

## 4. 资源管理

### 4.1 工作目录生命周期

```python
# 创建
work_dir = tempfile.mkdtemp(prefix="agentflow_")

# 每个节点在子目录中工作
node_dir = os.path.join(work_dir, f"node_{node.id}")
os.makedirs(node_dir, exist_ok=True)

# 全部结束后清理
try:
    shutil.rmtree(work_dir)
except:
    pass  # 清理失败不阻塞响应
```

**降级：** 即使 `shutil.rmtree` 失败（如文件被锁定），也不会阻塞结果返回。

### 4.2 成本追踪

```python
all_results = []
for fut in as_completed(fut_to_node):
    ...
    all_results.append(node.to_dict())

# 响应中包含总成本
response = {
    "nodes": all_results,
    "total_cost": sum(r.get("cost", 0) for r in all_results),
    "total_duration": sum(r.get("duration", 0) for r in all_results),
}
```

即使部分节点失败，总成本和总耗时仍然被正确计算。

## 5. 降级场景实战

### 场景 1：deploy 模板不存在

```
1. 节点 profile = "deploy"
2. TemplateEngine.load("deploy")
3. deploy.yaml 不存在
4. → 自动加载 dev.yaml 作为降级
5. Agent 使用 dev 模板执行
6. 节点成功完成（但角色是"开发型"而非"部署型"）
```

### 场景 2：LLM API 全量超时

```
1. 工作流有 8 个节点，thread_pool 并行执行
2. 其中 3 个节点 LLM API 超时（timeout=120s）
3. 5 个节点正常完成（status=ok）
4. 3 个超时节点返回 {status: "timeout"}
5. 最终响应包含 5 个 ok + 3 个 timeout
6. 前端展示时，超时节点显示为橙色标记
```

### 场景 3：Agent 在开发过程中遇到代码错误

```
1. Agent 写入 Python 代码（write 工具）
2. 执行 python script.py（bash 工具）
3. 返回 SyntaxError
4. Agent 在 Observe 阶段读取错误输出
5. 下一轮 Think 决定修复代码
6. 重新写入修复后的代码
7. 再次执行成功
8. 这是 Agent 的"自我修复"能力，在单次节点执行内完成
```

### 场景 4：工作目录磁盘空间不足

```
1. tempfile.mkdtemp() 抛出 OSError
2. 外层 try/except 捕获异常
3. handle_execute() 返回 500 Internal Server Error
4. 用户可重试请求
5. 建议：监控磁盘使用情况，保留至少 1GB 可用空间
```

## 6. 配置参考

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `timeout_s` (analysis) | 120 | 分析型节点超时 |
| `timeout_s` (design) | 120 | 设计型节点超时 |
| `timeout_s` (dev) | 180 | 开发型节点超时 |
| `timeout_s` (test) | 180 | 测试型节点超时 |
| `timeout_s` (doc) | 120 | 文档型节点超时 |
| `timeout_s` (deploy) | 300 | 部署型节点超时 |
| `max_turns` | 10–15 | 最大 Think→Act 轮数 |
| `OUTPUT_MAX_LENGTH` | 2000 | 结果输出最大字符数 |
| `template_fallback` | `dev.yaml` | 模板降级目标 |
| `work_dir` | `tempfile.mkdtemp()` | 临时工作目录 |

## 7. 最佳实践

1. **总是有兜底**：每个可能失败的操作都有降级路径（模板、LLM、变量）
2. **隔离错误**：节点级隔离确保单点故障不扩散
3. **幂等设计**：Agent 的写操作应该可重入
4. **超时分层**：不同粒度的超时防止单点卡死全局
5. **保留结果**：即使部分节点失败，也要返回所有节点的状态
6. **前端适配**：前端应能处理 `error`/`timeout` 状态，展示友好提示
