# 05.2 — 错误处理与重试策略

## 1. 概述

Orchestrator 的错误处理机制确保工作流在面对节点执行异常、LLM API 故障、超时等问题时能够自动恢复。整个流程基于 **节点级错误隔离** 原则——单个节点的失败不影响同组其他节点，也不影响整个工作流。

```
节点执行开始
    │
    ▼
┌──────────────────────────────────────┐
│  AgentRunner.execute()                │
│                                       │
│  ┌──────┐   ┌──────┐   ┌─────────┐   │
│  │ Think│──►│ Act  │──►│ Observe │   │
│  └──────┘   └──────┘   └─────────┘   │
│       │          │            │       │
│       └──────────┴────────────┘       │
│                  │                    │
│          ┌───────┴────────┐           │
│          │                │           │
│          ▼                ▼           │
│    ┌──────────┐    ┌──────────┐      │
│    │ 正常完成 │    │ 异常抛出 │      │
│    └──────────┘    └────┬─────┘      │
│                         │            │
└─────────────────────────┼────────────┘
                          │
                          ▼
                 ┌──────────────────┐
                 │  fut.result()    │
                 │  捕获异常        │
                 └──────┬───────────┘
                        │
                        ▼
              ┌─────────────────────┐
              │  错误结果封装        │
              │  {status: "error",  │
              │   result: "异常信息"}│
              └─────────────────────┘
                        │
                        ▼
              节点状态更新为 ERROR
              不影响同组其他节点
```

## 2. 错误分类

### 可恢复错误

| 错误类型 | 来源 | 默认行为 |
|----------|------|---------|
| **LLM API 超时** | LLM 调用无响应超过 `timeout_s` | 返回 `status: "timeout"`，节点标记为 error |
| **LLM 返回异常** | API 返回 5xx / 无效响应 | 返回 `status: "error"`，节点标记为 error |
| **工具执行失败** | bash/read/write 工具返回非零退出码 | 记录到 Observe 阶段，Agent 可能自行修复 |
| **资源不足** | 磁盘空间 / 内存不足 | 返回 `status: "error"`，节点标记为 error |

### 不可恢复错误

| 错误类型 | 来源 | 默认行为 |
|----------|------|---------|
| **WorkflowJSON 校验失败** | 节点 ID 不存在 / 自环 | 编译阶段即报错（400 Bad Request） |
| **循环依赖** | DAG 中存在环 | `topological_sort` 返回不完整 → 报错 |
| **模板加载失败** | `templates/` 目录下无 `dev.yaml` | 回退到空模板，继续执行 |
| **请求格式错误** | JSON 解析失败 | 返回 400 Bad Request |

## 3. 超时控制

### 节点级超时

每个节点的超时时间从 YAML 模板的 `timeout_s` 字段读取：

| 模板 | timeout_s | 说明 |
|------|-----------|------|
| `analysis.yaml` | 120 | 2 分钟 |
| `design.yaml` | 120 | 2 分钟 |
| `dev.yaml` | 180 | 3 分钟 |
| `test.yaml` | 180 | 3 分钟 |
| `doc.yaml` | 120 | 2 分钟 |
| `deploy.yaml` | 300 | 5 分钟 |

### AgentRunner 超时实现

```python
result = agent.execute(
    prompt=prompt,
    work_dir=node_dir,
    profile=node.profile,
    tools_enabled=True,
    max_turns=task.max_turns,
    timeout=task.timeout_s,   # ← 模板定义的超时时间
)
```

`AgentRunner` 内部在每次 Think→Act→Observe 循环中检查：
- 总运行时间是否超过 `timeout_s`
- 达到超时则停止循环，返回 `status: "timeout"`

### 空闲超时

如果 Agent 连续多轮无有效输出（如 LLM 无限自我对话），`max_turns` 会限制最大轮数：

| 模板 | max_turns | 说明 |
|------|-----------|------|
| `analysis.yaml` | 10 | 分析型通常迭代少 |
| `design.yaml` | 10 | 设计型 |
| `dev.yaml` | 15 | 开发型需要更多轮（编码+调试） |
| `test.yaml` | 15 | 测试型 |
| `doc.yaml` | 10 | 文档型 |
| `deploy.yaml` | 15 | 部署型 |

## 4. 重试机制

### 当前实现

当前的 `handle_execute()` 使用简单的"一次执行，异常兜底"策略：

```python
for fut in as_completed(fut_to_node):
    node = fut_to_node[fut]
    try:
        result = fut.result()
    except Exception as e:
        result = {
            "status": "error",
            "result": f"执行异常: {e}",
            "cost": 0, "duration": 0, "turns": 0,
        }
    # 无论成功还是失败，都更新节点状态
    node.status = result.get("status", "error")
    ...
```

**设计特点：**
- 一次执行，不自动重试
- 异常被捕获后封装为错误结果
- 节点状态标记为 `error`
- **不影响同组其他节点的执行**

### 可扩展的重试策略（推荐）

以下重试策略可以作为未来扩展：

#### 策略 1：快速重试

```python
MAX_RETRIES = 2
for attempt in range(MAX_RETRIES + 1):
    try:
        result = agent.execute(...)
        break
    except Exception as e:
        if attempt < MAX_RETRIES:
            time.sleep(2 ** attempt)  # 指数退避: 1s, 2s
            continue
        result = error_result(e)
```

#### 策略 2：选择性重试

仅对特定异常类型重试（如网络超时），对校验错误不重试：

| 异常类型 | 是否重试 | 重试间隔 |
|----------|---------|---------|
| `TimeoutError` | ✅ | 指数退避 |
| `ConnectionError` | ✅ | 指数退避 |
| `ValueError` | ❌ | — |
| `FileNotFoundError` | ❌ | — |

#### 策略 3：降级执行

```python
# 首次使用高精度模型
agent = AgentRunner(model="deepseek-v4")
result = agent.execute(...)

# 失败后降级到低成本模型
agent = AgentRunner(model="glm-4-flash")
result = agent.execute(...)
```

## 5. 错误状态流转

### 节点状态

```
PENDING ──→ READY ──→ RUNNING ──→ OK
                │            │
                │            └──→ ERROR
                │                  │
                └──────────────────┘
                        (校验失败)
```

| 状态 | 含义 |
|------|------|
| `pending` | 初始状态，等待执行 |
| `ready` | 已就绪，等待调度 |
| `running` | 正在执行 |
| `ok` | 执行成功 |
| `error` | 执行异常 |
| `timeout` | 执行超时 |
| `skipped` | 被跳过 |

### 错误信息的传递

```python
result = {
    "status": "error",          # ok / error / timeout
    "result": "执行异常: ...",   # 人类可读的错误描述
    "cost": 0,                  # 失败不计费
    "duration": 0,              # 耗时
    "turns": 0,                 # 尝试轮数
}
```

前端根据 `node.status` 渲染不同样式：
- `ok` → 绿色
- `error` → 红色
- `timeout` → 橙色
- `pending` → 灰色

## 6. 常见异常场景处理

### 场景 1：LLM API 临时不可用

```
1. AgentRunner 调用 LLM API 超时
2. ThreadPoolExecutor 捕获 TimeoutError
3. 封装结果：{status: "timeout", result: "LLM API 超时"}
4. 节点标记为 error
5. 同组其他节点继续执行
```

### 场景 2：单节点代码执行报错

```
1. Agent 写入代码文件（write 工具）
2. 执行代码（bash 工具）返回非零退出码
3. Agent 在 Observe 阶段看到错误输出
4. Agent 自行修复并重新执行（Think→Act→Observe 循环内）
5. 如果 max_turns 耗尽仍未修复 → 返回 error
```

### 场景 3：DAG 中存在循环依赖

```
1. topological_sort() 检测到入度无法归零
2. 排序结果不完整（sorted_ids < len(nodes)）
3. 返回的 parallel_groups 不包含环中的节点
4. 建议：编译阶段即应检测并拒绝
```

### 场景 4：模板文件缺失

```
1. TemplateEngine.load("custom_type")
2. custom.yaml 不存在
3. 自动回退到 dev.yaml
4. 使用 dev 模板渲染节点
5. 工作流继续执行，不中断
```

## 7. 配置参考

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `timeout_s` (per template) | 120–300 | 节点超时秒数 |
| `max_turns` (per template) | 10–15 | 最大交互轮数 |
| `MAX_RETRIES` | 0（当前） | 自动重试次数 |
| `RETRY_BACKOFF` | 2s（建议） | 重试退避基数 |

## 8. 最佳实践

1. **幂等设计**：Agent 的每个操作应可重入，多次执行结果相同
2. **渐进式超时**：简单节点用 120s，复杂节点用 300s，避免一刀切
3. **友好错误消息**：`result` 字段包含人类可读的错误描述，方便前端展示
4. **不中断同组**：一个节点失败不应 kill 整个线程池
5. **降级优先**：模板加载失败时回退到 dev.yaml，而不是直接报错
