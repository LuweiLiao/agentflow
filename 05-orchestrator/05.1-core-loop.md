# 05.1 — Orchestrator 核心循环

## 1. 概述

核心循环（Core Loop）是 Orchestrator 的心脏。它接收编译器产出的 `PromptTask[]` 数组，按依赖拓扑决定执行顺序，驱动 Docker 运行时，收集结果，并在失败时执行恢复策略。

```
                        ┌──────────────────────┐
                        │   Compiled            │
                        │   PromptTask[]        │
                        └──────────┬───────────┘
                                   │
                                   ▼
                        ┌──────────────────────┐
         ┌─────────────│  ① 拓扑遍历排序       │─────────────┐
         │             └──────────────────────┘             │
         │                                                  │
         ▼                                                  ▼
  ┌─────────────────┐                            ┌─────────────────┐
  │  ② 并行启动容器  │                            │  串行执行        │
  │  (parallel_group)│                            │  (独立节点)      │
  └────────┬────────┘                            └────────┬────────┘
           │                                              │
           └──────────────┬───────────────────────────────┘
                          │
                          ▼
               ┌──────────────────────┐
               │  ③ Docker 容器启动    │
               │  docker run ...       │
               └──────────┬───────────┘
                          │
                          ▼
               ┌──────────────────────┐
               │  ④ 运行时变量注入     │
               │  upstream_output     │
               │  → {prompt}          │
               └──────────┬───────────┘
                          │
                          ▼
               ┌──────────────────────┐
               │  ⑤ 监控 stdout       │
               │  解析 Envelope JSON   │
               │  超时检测             │
               └──────────┬───────────┘
                          │
                          ▼
               ┌──────────────────────┐
               │  ⑥ 输出校验          │
               │  output_schema       │
               └──────────┬───────────┘
                          │
                 ┌────────┴────────┐
                 ▼                 ▼
        ┌────────────────┐  ┌────────────────┐
        │  ⑦a 成功       │  │  ⑦b 失败       │
        │  保存结果       │  │  重试 / 回滚   │
        │  检查是否完成   │  │                │
        │  同步画布       │  │                │
        └────────────────┘  └───────┬────────┘
                 │                  │
                 │                  ▼
                 │         ┌────────────────┐
                 │         │  ⑧ 反馈       │
                 │         │  测试失败 →    │

                 │         │  新容器重试    │

                 │                  │

                 ALL NODES DONE
                 ──────────────
                 Result[] → 画布同步
```

## 2. 步骤详解

### 步骤 ①：接收 PromptTask[]，按 sequence 拓扑遍历

**输入：** 编译器产出的 `PromptTask[]` 数组，每个元素包含 `sequence`、`parallel_group`、`dependencies` 等元信息。

**处理逻辑：**

```
tasks = compiler_output.tasks

# 1. 构建邻接表 + 入度表
graph = {task.id: task.dependencies for task in tasks}
in_degree = {task.id: 0 for task in tasks}
for task in tasks:
    for dep in task.dependencies:
        in_degree[task.id] += 1

# 2. 拓扑排序（Kahn 算法）
queue = [t for t in tasks if in_degree[t.id] == 0]
sorted_tasks = []

while queue:
    # 同 parallel_group 的节点视为同一"批次"
    batch = extract_parallel_group(queue)
    sorted_tasks.append(batch)
    for task in batch:
        for dependent in task.dependents:
            in_degree[dependent.id] -= 1
            if in_degree[dependent.id] == 0:
                queue.append(dependent)
```

**关键约束：**

- `sequence` 数值决定层级，同层且无依赖的节点可并行
- `parallel_group` 相同的节点视为同一个并行批次
- 若检测到循环依赖，立即中止并抛出 `CycleDependencyError`

### 步骤 ②：串行/并行执行

根据拓扑排序结果确定执行模式：

| 条件 | 模式 | 说明 |
|------|------|------|
| `parallel_group == ""` 且无依赖 | 串行 | 单独执行，不与其他 Task 并发 |
| `parallel_group == "group-a"` | 并行 | 同组所有 Task 同时创建容器 |
| `sequence` 差 > 0 | 串行等待 | 前一层全部完成后才进入下一层 |

**并行执行策略：**

```
# 同 parallel_group 的节点并发
executor = ThreadPoolExecutor(max_workers=CONCURRENCY_LIMIT)
futures = []
for task in parallel_batch:
    future = executor.submit(execute_task, task, context)
    futures.append(future)

# 等待所有并行的 Task 完成
results = [f.result() for f in as_completed(futures)]
```

> 注意：并行度受 `CONCURRENCY_LIMIT`（默认 8）控制，防止宿主机资源耗尽。

### 步骤 ③：Docker 容器启动

为每个 PromptTask 启动一个独立的 Docker 容器。

```
docker run \
  --rm \
  -e HERMES_API_KEY={token} \
  -e TASK_ID={task.id} \
  -e SEQUENCE={task.sequence} \
  -v {work_dir}:/workspace:ro \
  claude:latest \
  claude --print "{prompt}"
```

参数说明见 [05.2-container-lifecycle.md](./05.2-container-lifecycle.md)。

### 步骤 ④：运行时注入

将上游 Task 的实际输出注入到当前 Task 的 prompt 模板中。

**注入位置标记：** `{upstream_output}` 或 `{{upstream.TASK_ID.output}}`

**注入流程：**

```
def inject_upstream_output(prompt_template, upstream_results):
    """
    将 upstream_results 中的 payload.data 替换到模板中。
    
    示例：
      模板: "请根据以下数据总结：{upstream_output}"
      upstream_results: {"task-001": Envelope{payload: {data: "..."}}}
      结果: "请根据以下数据总结：...（上游实际输出）"
    """
    for placeholder, result in upstream_results.items():
        data = result.payload.data
        prompt_template = prompt_template.replace(
            f"{{upstream.{placeholder}.output}}", data
        )
        prompt_template = prompt_template.replace(
            "{upstream_output}", data
        )
    return prompt_template
```

**注意：** 多个上游输出时使用命名占位符 `{upstream.TASK_ID.output}` 避免歧义。

### 步骤 ⑤：监控 stdout，解析 Envelope JSON

容器启动后，Orchestrator 实时读取 stdout，逐行解析标准格式的 JSON 行（每行一个 Envelope）。

**Envelope 格式：**

```json
{"type": "chunk", "seq": 1, "payload": {"data": "部分输出...", "mime": "text/plain"}}
{"type": "chunk", "seq": 2, "payload": {"data": "继续输出...", "mime": "text/plain"}}
{"type": "final", "seq": 3, "payload": {"data": "完整结果", "mime": "text/plain", "metadata": {...}}}
{"type": "error", "payload": {"code": "PARSE_ERROR", "message": "..."}}
```

**监控逻辑：**

```
container = docker_client.containers.run(image, command, **run_kwargs)

full_output = []
timeout = task.timeout_seconds or DEFAULT_TIMEOUT  # 默认 300s

for line in container.logs(stream=True, timestamps=False):
    line = line.decode('utf-8').strip()
    
    try:
        envelope = json.loads(line)
    except json.JSONDecodeError:
        # 非 JSON 行：可能有副本文本，记录为原始输出
        full_output.append(("raw", line))
        continue
    
    if envelope["type"] == "chunk":
        full_output.append(("chunk", envelope["payload"]["data"]))
        
    elif envelope["type"] == "final":
        final_envelope = envelope
        break
        
    elif envelope["type"] == "error":
        raise TaskExecutionError(envelope["payload"])
        
    # 更新超时计时器
    if time.monotonic() - start_time > timeout:
        container.kill()
        raise TimeoutError(f"Task {task.id} 超时 ({timeout}s)")
```

**超时处理：** 超过 `task.timeout_seconds` 后调用 `docker kill` 强制终止容器，然后按错误恢复流程处理。

### 步骤 ⑥：output_schema 校验

使用 PromptTask 定义的 `output_schema` 对最终输出的 `payload.data` 进行校验。

**支持格式：**

- **JSON Schema**：校验 JSON 结构
- **TypeScript 类型断言**：运行时正则匹配
- **自定义校验器**：通过 `validator` 字段指定

```
def validate_output(envelope, schema):
    if not schema:
        return True  # 无 schema 默认通过
    
    data = envelope["payload"]["data"]
    
    if schema.get("type") == "json":
        # JSON Schema 校验
        import jsonschema
        try:
            json_data = json.loads(data)
            jsonschema.validate(json_data, schema["schema"])
            return True
        except (json.JSONDecodeError, jsonschema.ValidationError) as e:
            raise ValidationError(str(e))
    
    elif schema.get("type") == "regex":
        # 正则匹配
        if not re.match(schema["pattern"], data):
            raise ValidationError(f"输出不匹配模式: {schema['pattern']}")
        return True
    
    return True
```

### 步骤 ⑦b：失败——重试 / on_failure 策略

执行失败时，根据 PromptTask 配置的策略处理：

**策略表：**

| `on_failure` | 行为 |
|-------------|------|
| `"retry"` | 按照 `retry.count` 重试次数重新执行当前 Task |
| `"retry_delay"` | 重试前等待 `retry.delay_seconds` |
| `"skip"` | 跳过当前 Task，标记为 `skipped`，继续下游 |
| `"abort"` | 立即中止整个工作流 |
| `"rollback"` | 回滚到上一个检查点（见 05.3） |

```
def handle_failure(task, error, attempt, max_retries):
    if attempt < max_retries:
        delay = task.retry.delay_seconds
        logger.warning(f"Task {task.id} 失败 (第 {attempt+1}/{max_retries} 次), "
                      f"{delay}s 后重试...")
        time.sleep(delay)
        return execute_task(task, context)  # 重新执行
    else:
        strategy = task.on_failure or "abort"
        if strategy == "skip":
            return TaskResult(task_id=task.id, status="skipped")
        elif strategy == "abort":
            raise WorkflowAborted(f"Task {task.id} 重试耗尽，工作流中止")
        elif strategy == "rollback":
            return rollback_to_checkpoint(task)
```

### 步骤 ⑧：反馈——测试失败 → 重编译上游 → 新容器重试

当 Task 执行成功但**下游测试**失败时（例如验证 Agent 发现输出不符合预期），触发反馈循环。

**触发条件：** 下游验证 Task 对当前 Task 的输出执行测试，测试结果为 `fail`。

**反馈流程：**

```
def feedback_retry(failed_task, test_result):
    """
    failed_task: 测试失败的 Task
    test_result: 包含测试失败详情
    """
    # 1. 标记当前 Task 为需重试
    failed_task.status = "needs_retry"
    
    # 2. 定位上游 Task（即 failed_task 的直接依赖）
    upstream_tasks = failed_task.dependencies
    
    # 3. 重新编译上游 Task（调整 prompt 或参数）
    for up_task in upstream_tasks:
        adjusted_prompt = adjust_prompt(
            up_task.prompt_template, 
            test_result.feedback
        )
        up_task.prompt_template = adjusted_prompt
        
        # 4. 为上游 Task 启动新容器重试
        new_result = execute_task(up_task, context)
        
        # 5. 用新结果重新执行当前 Task
        if new_result.status == "success":
            return execute_task(failed_task, context)
    
    raise FeedbackRetryError("上游重编译后仍无法通过测试")
```

**关键设计点：**

- 反馈循环默认最大深度为 3，防止无限循环
- 每次反馈迭代都会记录到 WAL，支持断点回溯
- 可选的 LLM-as-Judge 模式：用另一个 Agent 判断输出质量，决定是否触发反馈

## 3. 核心循环伪代码总览

```python
def orchestrator_core_loop(tasks: list[PromptTask]):
    sorted_batches = topological_sort(tasks)
    results = {}
    
    for batch in sorted_batches:
        # 并行批次
        with ThreadPoolExecutor(max_workers=CONCURRENCY_LIMIT) as pool:
            futures = {}
            for task in batch:
                upstream_data = collect_upstream(task.dependencies, results)
                future = pool.submit(execute_single_task, task, upstream_data)
                futures[future] = task
            
            for future in as_completed(futures):
                task = futures[future]
                try:
                    result = future.result()
                    results[task.id] = result
                except Exception as e:
                    result = handle_failure(task, e, ...)
                    results[task.id] = result
                    if result.status in ("aborted", "failed"):
                        raise WorkflowAborted()
    
        # 同步画布
        sync_canvas(results)
    
    return results


def execute_single_task(task, upstream_data):
    # 步骤 ④：注入
    prompt = inject_upstream(task.prompt_template, upstream_data)
    
    # 步骤 ③：启动容器
    container = start_container(task.image, prompt, task.env_vars)
    
    # 步骤 ⑤：监控 + 解析
    envelope = monitor_container(container, task.timeout)
    
    # 步骤 ⑥：校验
    validate_output(envelope, task.output_schema)
    
    # 步骤 ⑦a：保存
    return TaskResult(task_id=task.id, status="success", envelope=envelope)
```

## 4. 配置参考

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `CONCURRENCY_LIMIT` | 8 | 最大并行容器数 |
| `DEFAULT_TIMEOUT` | 300 | 单个 Task 默认超时（秒） |
| `MAX_FEEDBACK_DEPTH` | 3 | 反馈循环最大迭代次数 |
| `WAL_PATH` | `/var/agentflow/wal/` | WAL 日志存储路径 |
| `STREAM_BUFFER_SIZE` | 4096 | stdout 读取缓冲区（字节） |

## 5. 注意事项

1. **幂等性**：每个 Task 应设计为幂等执行，确保重试不产生副效应
2. **资源回收**：无论成功或失败，容器必须清理（`--rm` 或显式 `docker rm`）
3. **日志完整性**：所有 Envelope 行和原始 stdout 均需持久化到 WAL
4. **拓扑不变性**：执行期间不允许修改 task 的依赖关系
